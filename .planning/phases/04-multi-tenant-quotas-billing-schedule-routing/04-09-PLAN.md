    # Runbook — Phase 4 Multi-Tenant Quotas, Billing & Schedule Routing

    ## At-a-Glance Severity Matrix

    | Symptom                                          | Likely cause                                                      | First response                                            |
    |--------------------------------------------------|-------------------------------------------------------------------|-----------------------------------------------------------|
    | 429 rate_limit_rps surge for one tenant          | Tenant burst exceeds rps_limit                                    | `gatewayctl tenant set-quota --tenant X --rps 50` (temporary) |
    | 429 quota_exceeded_daily_tokens for one tenant   | Daily allocation exhausted                                        | Raise `--daily-tokens` OR wait for 00:00 BRT rollover     |
    | 503 quota_check_unavailable                      | Postgres usage_counters lookup failed                             | Check Postgres health; fail-closed was intentional D-A2   |
    | 503 off_hours_upstream_unavailable               | Tenant in peak mode off-hours AND OpenRouter/Fireworks down       | Check Fireworks status; decision D-C2 is no-fallback     |
    | Cost numbers wrong in /admin/usage               | Seed price drift OR fx stale                                      | `gatewayctl prices list` + `prices set` to correct        |
    | billing_events growing but usage_counters stale  | Flusher goroutine dropping OR DB contention                       | Check `gateway_billing_flush_failures_total`; reconcile   |

    ### Adjust a tenant's RPS/RPM + daily/monthly quota

    docker exec ai-gateway-dev /gatewayctl tenant set-quota \
      --tenant converseai \
      --rps 50 --rpm 1500 \
      --daily-tokens 20000000 --monthly-tokens 600000000 \

      --daily-embeds 200000 --monthly-embeds 6000000

    ## Incident Playbooks

    ### Incident: one tenant hammering rate-limit → legitimate requests rejected

    Symptoms: `gateway_rate_limit_rejected_total{tenant=X}` spikes; tenant complains.

    1. Verify via `/metrics`: confirm which window (rps vs rpm) is failing.
    2. Check if burst is legitimate (e.g., overnight batch) — if so, raise RPS/RPM via `gatewayctl tenant set-quota`.
    3. If burst is unexpected (possible runaway client loop), contact the tenant owner.
    4. Do NOT disable the rate-limit globally. Fail-open ONLY on Redis transport error (cfg.RateLimitFailOpen=true). If the rate-limit itself is working but too aggressive, raise the limit per-tenant.

    ### Incident: quota_check_unavailable 503s (fail-closed)

    Symptoms: `gateway_quota_check_failures_total` incrementing; users see 503.

    1. Check Postgres health: `docker logs infra-postgres | tail -100`; `pg_isready -h <host>`
    2. Check gateway logs for the specific query failing (`GetUsageCountersToday` vs `GetUsageCountersMonth`).
    3. If Postgres is transient-degraded: wait; fail-closed is the correct behavior. Refusing to serve = cheaper than torched external cost.
    4. If Postgres is down for >5 min: consider flipping `AI_GATEWAY_QUOTA_FAIL_OPEN=true` in Portainer + restart. **This is an emergency override; reset to `false` within an hour.** Document in incident log.

    ### Incident: billing_events table growing but usage_counters lagging

    Symptoms: `/admin/usage` response shows older totals than `gatewayctl billing reconcile` reports.

    1. Check `gateway_billing_flush_dropped_total` counter — non-zero means flusher channel back-pressure.
    2. Check `gateway_billing_flush_failures_total{reason}` — non-zero means DB writes failing.
    3. Check Postgres deadlock log for usage_counters UPSERT contention (Pitfall 2).
    4. Run `gatewayctl billing reconcile --apply` to rewrite usage_counters from authoritative events.
    5. If flusher is chronically overwhelmed, the next iteration is per-tenant flush batching (Pitfall 2 mitigation) — file a Phase 5 ticket.

    ### Incident: 503 off_hours_upstream_unavailable

    Symptoms: Peak-mode tenant with a chat request at off-hours gets 503.

    1. Check OpenRouter/Fireworks status: https://fireworks.ai/status (or operator's preferred)
    2. Check `gateway_breaker_state{upstream=openrouter-chat}` — if OPEN, external provider is unreachable.
    3. Decision is intentional (D-C2 + Phase 3 D-C4): no fall-of-fallback to OpenAI direct chat — keeps Qwen model fixed.
    4. Affected tenant: temporarily flip to `mode=24/7` to restore service (`gatewayctl tenant set-mode --tenant X --mode 24/7`) — the GPU may be running.
    5. After provider recovers: flip back to peak mode.

    ### Incident: gatewayctl prices set fails with "unknown provider" at cost-calc time

    Symptoms: billing_events rows have cost_external_brl=0 despite upstream being tier-1.

    1. `gatewayctl prices list` — confirm there's an ACTIVE row for the (model, provider, unit) combo matching what the dispatcher is using.
    2. The provider string MUST match exactly what the dispatcher records in billing_events.upstream (e.g., `openrouter-fireworks`, not `openrouter` or `fireworks`).
    3. If missing, add with `gatewayctl prices set ...`.
    4. Reconcile is best-effort: it cannot recompute cost on historical rows where prices were missing at flush time; those rows stay with cost=0. Accept the gap; document in incident log.

    ## Cross-Replica Coherence — Deferred to Phase 6

    Today (Phase 4) the gateway runs as a SINGLE replica in dev. When Phase 6
    ships the auto-provisioning emergency pod, we'll have up to 2 gateway
    replicas simultaneously. At that point:

    - Redis Lua atomic rate-limit → correct across replicas (Redis is single-threaded).
    - Postgres quota counter → correct across replicas (PG row lock).
    - In-memory config snapshots (tenants, prices, fx, upstreams) → each replica
      refreshes independently on NOTIFY. Typical staleness between replicas: <2s.
    - Admin key cache → each replica has its own 60s cache.

    No special handling required in Phase 4.
