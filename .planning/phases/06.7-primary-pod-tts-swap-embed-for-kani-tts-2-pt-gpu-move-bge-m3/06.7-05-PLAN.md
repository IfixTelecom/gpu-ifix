<voice_persistence_contract>
## Voice-Embedding Persistence Contract (REVIEWS consensus action #1 — HIGH, both reviewers)

Vast.ai pod storage is EPHEMERAL. A speaker embedding `.pt` cached only on the pod is LOST on pod replacement. The managed-voice flow therefore has ONE source of truth in durable storage and a rebuildable local cache:

```
voice_id  ─┬─►  Postgres ai_gateway.voices row   (id=voice_id, tenant_id, label, s3_key)   [DURABLE — Plan 04/07]
           └─►  MinIO S3 object  s3_key = <S3_VOICE_PREFIX>/<voice_id>.wav  (reference WAV)  [DURABLE — written by Plan 07 upload]

          ┌─►  pod-local cache  <KANI_SPEAKERS_DIR>/<voice_id>.pt  (speaker embedding)        [EPHEMERAL — rebuildable]
```

Generation mechanism = LAZY (both reviewers' endorsed choice):
- The gateway upload (Plan 07) persists ONLY the reference WAV to S3 + writes the catalog row. It does NOT push a `.pt`.
- On a `/v1/audio/speech` request with a `voice_id` whose `.pt` is NOT in the local speakers dir (cache miss — fresh pod, evicted cache), the shim:
  1. fetches `<S3_VOICE_PREFIX>/<voice_id>.wav` from MinIO (creds from env, see config below),
  2. runs `embedder.embed_audio_file(wav)` to regenerate the speaker embedding,
  3. caches it to `<KANI_SPEAKERS_DIR>/<voice_id>.pt`,
  4. proceeds with synth.
- On cache hit, use the local `.pt` directly (no S3 round-trip).
- If the S3 object does not exist for that `voice_id` → 404 (unknown voice).

This makes a voice survive pod replacement: cache miss → refetch from S3 → regenerate → cache → synth. `test_voice_cache_miss_refetches_from_s3` asserts exactly this.
</voice_persistence_contract>

## Config env vars (REVIEWS consensus action #6 — make limits/paths explicit, not hardcoded)

Shim reads these from env (with defaults), NOT hardcoded constants:
- `KANI_MODEL_CACHE_DIR` (HF/torch model cache; default e.g. /opt/kani-data/models)
- `KANI_SPEAKERS_DIR` (local .pt cache; default e.g. /opt/kani-data/speakers)
- `KANI_S3_VOICE_PREFIX` (S3 key prefix for reference WAVs; e.g. voices)
- `KANI_MAX_INPUT_CHARS` (max synth text length; DoS cap, e.g. 4000)
- `KANI_MAX_UPLOAD_BYTES` (max reference WAV bytes if the shim ever receives one directly; align with gateway MaxBytes)
- MinIO creds for the S3 fetch: reuse the pod's existing MinIO env (MINIO_ENDPOINT / MINIO_ACCESS_KEY / MINIO_SECRET_KEY / MINIO_BUCKET — already forwarded to the pod onstart per lifecycle.go env map).
- Kani weights env: the model weights key/sha forwarded into the pod (names per WAVE0-GATES / config; mirror the prior PRIMARY_BGEM3_WEIGHTS_* shape that embed used).

The gateway-side config (UPSTREAM_TTS_URL, UPSTREAM_TTS_PIPER_URL, gateway max-text/max-upload) is enumerated in Plan 07.
</config_surface>
