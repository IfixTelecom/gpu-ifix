# Cross-AI Plan Review — Phase 06.7

> Self-review (claude) skipped — running inside Claude Code. Reviewed by Gemini + Codex for independence.

## Gemini Review

The implementation plans for Phase 06.7 are exceptionally well-aligned with the existing architectural standards and patterns of the `ifix-ai-gateway` project. They systematically address the dual goals of enhancing the system's capabilities (GPU-based TTS with voice cloning) and resolving significant technical debt (24/7 RAG availability and primary pod override resilience).

### Summary
The plans provide a high-fidelity roadmap for integrating GPU-based TTS and relocating the embedding service. The strategy effectively leverages the project's mature role-agnostic routing and provisioning systems. By front-loading a "Wave 0" for empirical validation of the TTS engine and establishing a rigorous test-first approach, the plans minimize the risk of late-stage failures. The architectural shift to move embeddings to a 24/7 CPU host is a critical improvement that ensures RAG stability during off-peak windows.

### Strengths
- **Empirical Validation (Wave 0):** Plan 01's focus on "Gate" decisions for Kani-TTS-2-pt's Portuguese quality and CUDA compatibility on a live 5090 is a vital defensive measure.
- **Nyquist Rigor:** Plan 02's creation of RED test stubs for every new behavior (proxy, probe, quota, reconciler) ensures high validation standards and architectural compliance.
- **Architectural Reuse:** The plans skillfully reuse the `DispatcherConfig`, `ReverseProxy`, and `OverrideTier0` patterns, maintaining a lean and consistent codebase.
- **Resilience Engineering:** The fix for **Pitfall #11** (Plan 08) proactively solves a known race condition where emergency cutbacks could leave the primary pod's routing in a stale state.
- **Multi-tenant Security:** The voice catalog implementation (Plan 07) correctly enforces isolation by sourcing `tenant_id` exclusively from the authenticated context, following ASVS V4 standards.

### Concerns
- **HIGH — Kani-TTS-2-pt Language Fitness:** As noted in **RESEARCH §Open Questions Q1**, the model card lists English as primary. While the `-pt` naming is promising, the voice-cloning prosody for Portuguese remains the primary risk. The Wave 0 gate is the correct mitigation, but this remains a potential phase-blocker.
- **MEDIUM — Piper Fallback Contract:** The fallback to 8kHz ulaw audio (Plan 07/GATE 3) represents a significant quality drop from 22kHz WAV. Some client applications may fail if they cannot handle the sample rate or codec transition.
- **MEDIUM — S3 Client Standardization:** Plan 07 introduces the first Go-based S3/MinIO interaction in the gateway. Without an existing pattern, there is a risk of inconsistent boilerplate for client initialization and error handling.
- **LOW — Image Size Overhead:** Adding the Kani stack (torch + nemo) to the primary pod image (Plan 05) increases the risk of exceeding the 40GB Vast.ai disk budget, though this is appropriately monitored in Gate 2.

### Suggestions
- **Standardize S3 Client Initialization:** Create a small `internal/s3x` package to wrap the AWS/MinIO client setup using existing `config.Minio*` variables to provide a clean pattern for the `voices` handler.
- **Audio Normalization Layer:** Consider adding an optional ffmpeg-based conversion interceptor in the gateway's Piper adapter to upsample ulaw to 16kHz WAV, providing a more stable contract for clients during fallbacks.
- **Warmup String Optimization:** In `kani_shim.py`, ensure the startup "warmup" synthesis uses a representative Portuguese string to verify that all phoneme-to-audio weights are fully resident in VRAM.
- **Corpus Dimension Alert:** Since Phase 8 will require a `pgvector` dimension migration (1536 → 1024), add a warning comment in the new `0025_create_voices.sql` migration to remind future developers of this upcoming change.

### Risk Assessment
**Risk Level: LOW**
**Justification:** The core logic is a straightforward extension of proven patterns. The most significant technical risks (VRAM budget, engine quality, and pod compatibility) are addressed early via autonomous test stubs and manual decision gates. The decoupling of the embedding service significantly improves the system's core value proposition.

## Codex Review

### Summary

The plan set is unusually thorough and generally coherent: it correctly front-loads the uncertain parts, separates gateway routing, schema, pod image, embed relocation, reconciler behavior, and live UAT into sensible waves, and keeps the key architectural correction intact: embed leaves the dynamic pod lifecycle and TTS enters it. The biggest risks are not in the high-level architecture, but in implementation details around voice-clone storage flow, Piper adapter behavior, Kani runtime compatibility, and the fact that several "autonomous" plans still depend on gate decisions or newly introduced interfaces that may not exist yet. Overall this is a strong plan, but it needs tightening around dependency gates, S3/voice embedding responsibilities, and fallback format compatibility.

### Strengths

- Clear phase boundary: TTS on GPU, embed on CPU 24/7, no corpus re-embed in this phase.
- Good Wave 0 discipline: Kani PT-BR quality, CUDA/image compatibility, Piper fallback format, and embed host sizing are validated before heavy implementation.
- Correct architectural move: `embed` removed from dynamic `OverrideTier0`; `tts` added because pod IP is dynamic.
- Pitfall #11 is handled explicitly and across all affected dynamic roles: `llm`, `stt`, `tts`.
- Security concerns are not bolted on late: tenant isolation, path traversal, upload size limits, and biometric data concerns are called out.
- Validation strategy is concrete: named tests, live UAT scenarios, and manual gates map well to the phase requirements.
- The plans avoid redesigning the gateway FSM or fallback chain, which keeps scope mostly contained.

### Concerns

#### HIGH

- **Voice clone flow is underspecified between gateway, MinIO, and Kani shim.** Plan 07 says upload persists WAV to MinIO and "triggers the shim's speaker-embedding extraction path," while Plan 05 says the shim resolves local `.pt` files. There is no clear mechanism for how a pod with ephemeral storage gets the uploaded WAV or generated `.pt`, how it survives pod replacement, or how `voice_id` maps to local speaker embeddings after a new pod lifecycle.
- **Piper fallback compatibility remains a major implementation risk.** The plans acknowledge Piper is not OpenAI-compatible and emits `ulaw 8kHz`, but Plan 07 leaves the adapter conditional on GATE 3. That is acceptable as a gate, but downstream implementation must be deterministic. If GATE 3 picks adapter, there needs to be an explicit plan for audio conversion, content-type negotiation, and `response_format` behavior.
- **Plan 05 may violate the "own thin shim" decision if it ports too much from the official server.** The plan says "port the shim from the official server" while D-05 says no community wrapper dependency and own shim. That is fine if used as reference, but the plan should state clearly that code/license review is required and only minimal implementation is copied or reimplemented.
- **Runtime storage for Kani model and voice embeddings could break across pod restarts.** The pod is Vast.ai lifecycle-managed. If speaker embeddings are only local files, any pod replacement loses them. The desired managed-voice contract implies either the shim fetches reference WAVs from MinIO and rebuilds/cache embeddings on demand, or the gateway/pod stores `.pt` embeddings in S3 too.

#### MEDIUM

- **Plan 02 "RED/skip stubs" may create false confidence.** Skipped tests satisfy compile-time scaffolding but do not fail if implementation never happens. Each downstream plan must explicitly unskip and make tests assert behavior as a hard acceptance criterion.
- **Plan 04 migration Down may fail if TTS rows are referenced.** Dropping `tts` from the CHECK after deleting seeded rows is okay only if no user-created `tts` rows remain. Probably acceptable for dev, but worth noting.
- **`tenant_id TEXT` may not match existing tenant identity semantics.** If existing auth context uses app IDs, API key IDs, or tenant IDs with stronger typing elsewhere, the voices schema should match that convention. Otherwise tenant isolation tests may pass while operational reporting/billing becomes awkward.
- **Config additions may be incomplete.** Plans mention `UPSTREAM_TTS_URL` and `UPSTREAM_TTS_PIPER_URL`, but also lifecycle/Kani weights env vars. The exact config surface for Kani model cache, speakers dir, max input length, upload max bytes, and S3 voice prefix should be explicit.
- **TTS quota classification may be too coarse.** A single `RouteClassTTS` may be enough for v1, but TTS cost scales by characters/audio duration, not request count. Can be deferred, but record the risk.
- **Embed service endpoint mismatch needs verification.** Research says Infinity serves `/embeddings`, while gateway route is `/v1/embeddings`. The static upstream must match the existing embed proxy expectations. A path rewrite may be required.
- **Kani dependency footprint may exceed disk/cold-start budgets.** Plan 01 catches this, but Plan 05 should explicitly require recording final image size and model-cache strategy after Dockerfile changes, not only during the spike.

#### LOW

- **Some plans overuse grep as verification.** Several acceptance criteria should rely more on tests/builds than string matching.
- **`response_format` support may be partial.** The phase only needs `wav|pcm`, but unsupported formats should return clean 400s.
- **Health endpoint for `kani_shim.py` is implied but not explicit.** Lifecycle probes use `/health` on port 8003. Plan 05 should explicitly implement `GET /health`.
- **LGPD risk is acknowledged but not operationalized.** Retention/deletion behavior for reference WAVs and speaker embeddings should be defined at least minimally.

### Suggestions

- Add an explicit design note before Plan 07/05 implementation: `voice_id -> Postgres row -> S3 WAV key -> optional S3 speaker embedding key -> pod local cache`. The shim should be able to rebuild or fetch the speaker embedding after pod replacement.
- Decide whether speaker embeddings are generated by: (1) gateway calling the Kani shim during voice upload, (2) shim lazily generating on first speech request after fetching the WAV, or (3) an async worker. For this phase, lazy generation plus local cache is probably simplest.
- Make `GET /health` a required shim endpoint and include it in Plan 05 tests.
- Make GATE 3 output a concrete implementation contract consumed by Plan 07: `gateway adapter`, `voice-api route`, or `controlled 503`. If adapter is chosen, add exact expected output format and conversion tool/library.
- Strengthen Plan 06 with a path-rewrite check: verify the gateway can call the CPU embed service through the existing `/v1/embeddings` route, not only direct `/embeddings`.
- In Plan 02, require skipped tests to become non-skipped in their owning implementation plan.
- Add explicit cleanup semantics for `DELETE /v1/audio/voices`: delete catalog row and S3 object; decide whether S3 delete failure is retryable, logged, or blocking.
- Add idempotency/duplicate behavior for voice uploads if clients retry after timeout.
- Add max text length and max audio upload size values as config, not hardcoded constants.
- Add one integration test for fallback path: tier-0 TTS fails or breaker opens, tier-1 Piper path receives the request.

### Per-Plan Notes

- **06.7-01** — Strong and necessary. Four gates are the right ones. Improvement: gate document should include exact downstream decisions, not just PASS/FAIL, especially Piper format and Kani image base. Risk: **MEDIUM** (depends on live GPU/operator validation, but mitigates higher downstream risk).
- **06.7-02** — Useful scaffolding, but skipped tests can hide missing implementation if not enforced later. Risk: **LOW-MEDIUM**.
- **06.7-03** — Good surgical plan. Removing `embed` from override map and adding `tts` is correct. Verify `Resolve("embed",0)` still works through static upstream. Risk: **LOW**.
- **06.7-04** — Schema direction sound. Biggest watchpoint is tenant identity consistency and cleanup semantics for voices. Risk: **MEDIUM**.
- **06.7-05** — Core risk plan. Kani shim, CUDA deps, model warmup, local speaker embedding, pod packaging all operationally sensitive. Add `/health` and clarify persistence/cache behavior for voice embeddings. Risk: **HIGH**.
- **06.7-06** — Good separation of embed serving from corpus migration. Must verify gateway compatibility with Infinity path and 1024-dim response through the gateway, not just direct curl. Risk: **MEDIUM**.
- **06.7-07** — Feature-heavy plan with most integration surface: proxy, dispatcher, config, S3, DB, auth, voices CRUD, Piper fallback. Achievable but needs sharper boundaries, especially voice embedding generation and Piper adaptation. Risk: **HIGH**.
- **06.7-08** — Well-targeted and important. Ready-tick reassert fix is pragmatic and low-latency. Main concern: ensure all old `embed` lifecycle references removed without breaking embed static routing. Risk: **MEDIUM**.
- **06.7-09** — Good live UAT coverage. Validates behaviors that cannot be proven locally. Add cleanup verification for MinIO voice objects and Vast instances. Risk: **MEDIUM**.

### Overall Risk Assessment

**Overall risk: MEDIUM-HIGH.** The architecture is sound and the plans are well sequenced, but the phase crosses several hard boundaries at once: GPU runtime packaging, dynamic pod lifecycle, binary audio proxying, voice-clone storage, S3/Postgres cataloging, and live fallback behavior. The riskiest areas are Kani runtime compatibility, voice embedding persistence across pod replacement, and Piper fallback format adaptation. With the suggested clarifications — especially around managed voice storage and fallback contract — the plan should achieve the phase goals.

## Consensus Summary

Both reviewers rate the phase architecturally sound with strong Wave 0 discipline, correct embed→CPU / TTS→GPU boundary, and good security hygiene. They diverge on overall risk: Gemini lands LOW (proven-pattern extension), Codex lands MEDIUM-HIGH (many hard boundaries crossed at once). The split is explained by scope of lens — Gemini weighted architectural fit, Codex weighted implementation-detail underspecification.

### Agreed Strengths
- Wave 0 empirical gates (Kani PT-BR quality, CUDA/image compat, Piper format, embed host sizing) front-load the real risk.
- Correct architectural move: `embed` out of dynamic `OverrideTier0`, `tts` in — driven by dynamic pod IP.
- Heavy reuse of proven patterns (`DispatcherConfig`, `ReverseProxy`, `OverrideTier0`); no FSM/fallback redesign.
- Pitfall #11 fix handled explicitly across all dynamic roles.
- Security not bolted on late: tenant isolation, path traversal, upload limits, biometric/LGPD flagged.

### Agreed Concerns (highest priority — raised by both)
1. **HIGH — Kani-TTS-2-pt Portuguese fitness** is the potential phase-blocker; Wave 0 gate is the correct (and only) mitigation.
2. **HIGH/MEDIUM — Voice-clone storage flow across pod replacement is underspecified.** Plan 07 (upload→MinIO + "trigger shim extraction") and Plan 05 (shim resolves local `.pt`) don't connect. Ephemeral Vast.ai storage means local-only speaker embeddings are lost on pod replacement. **Both want an explicit `voice_id → Postgres row → S3 WAV → optional S3 .pt → pod local cache` contract before Plan 05/07 execution.**
3. **MEDIUM — Piper fallback contract (GATE 3)** must become a deterministic implementation contract: audio conversion (ulaw 8kHz → WAV), content-type negotiation, `response_format` behavior. Gemini suggests an ffmpeg normalization layer.
4. **MEDIUM — S3/MinIO is the first Go-side S3 interaction** in the gateway; both implicitly want a standard client wrapper (Gemini explicit: `internal/s3x`).
5. **LOW/MEDIUM — Image/disk budget** for the Kani stack on the 40GB Vast.ai pod; Plan 05 should record final image size + model-cache strategy, not only Wave 0 spike.

### Divergent Views
- **Overall risk rating:** Gemini LOW vs Codex MEDIUM-HIGH. Treat Codex's rating as the working assumption — its concerns are concrete and actionable, and the gap is about unspecified implementation details rather than architectural disagreement.
- **RED/skip test stubs (Plan 02):** Codex flags as potential false confidence and wants a hard "unskip in owning plan" acceptance criterion; Gemini praises the same stubs as Nyquist rigor. Both reconcilable — adopt Codex's enforcement gate.

### Recommended Actions Before Execution
1. Resolve the voice-embedding persistence contract (lazy regen from S3 WAV + local cache is the simplest path both reviewers endorse).
2. Make GATE 3 emit a concrete Piper adapter contract for Plan 07.
3. Add hard acceptance criterion: each Plan 02 skipped test must be unskipped + asserting in its owning plan.
4. Add `GET /health` as a required `kani_shim.py` endpoint in Plan 05.
5. Verify embed path-rewrite (`/v1/embeddings` → Infinity `/embeddings`) in Plan 06.
6. Move max-text-length, max-upload-bytes, Kani cache/speakers dir, S3 voice prefix to explicit config.
