## Summary

Esta fase entrega DUAS coisas independentes que compartilham a mesma máquina de tier-0 do gateway. (1) **TTS GPU no primary pod** — um 5º child supervisord (`[program:kani]`) rodando um shim FastAPI fino que serve Kani-TTS-2 (`nineninesix/kani-tts-2-pt`, 400M params, backbone LiquidAI LFM2 350M + Nvidia NanoCodec, ~3 GB VRAM, RTF ~0.2, Apache 2.0) via `POST /v1/audio/speech` OpenAI-compatible, com voice-clone por speaker-embedding. (2) **Relocação do embed** para um host CPU 24/7 (n8n-ia-vm), conectado como upstream tier-0 estático via `UPSTREAM_EMBED_URL`, removendo `"embed"` do mapa `OverrideTier0` do reconciler. O gateway ganha: rota `/v1/audio/speech` (proxy JSON→binary próprio), CRUD `/v1/audio/voices` (catálogo multi-tenant em Postgres `ai_gateway` + WAV em MinIO), role `tts` no tier-0 override, probe `tts`, RouteClass `tts` na quota, e o fix do Pitfall #11 cobrindo llm/stt/tts juntos.

A arquitetura existente é extremamente favorável: o mapa `tier0Override` já é role-agnostic (loader.go:87-93 já tem `llm`/`stt`/`embed`), o probe é switch-by-role, a tabela `upstreams` segue padrão env-var, e há precedente direto (goose migrations, MinIO client no onstart, multi-stage Dockerfile com supervisord). O custo de engenharia está concentrado em (a) escrever o shim Kani, (b) escrever o proxy JSON→binary + handlers de voice CRUD, (c) o fix do Pitfall #11.

**Recomendação principal:** Embed model = **`intfloat/multilingual-e5-large`** servido por **Infinity** (mesma engine já no pod, `infinity-emb==0.0.77`) com **quantização ONNX int8** para caber confortável em CPU no n8n-ia-vm; mas como o corpus atual usa OpenAI 1536-dim e o re-embed é Phase 8, a escolha de modelo aqui é a do modelo a SERVIR (não bloqueia o corpus). TTS = Kani-TTS-2-pt via shim derivado do servidor oficial `nineninesix-ai/kani-tts-2-openai-server` (que já implementa exatamente o contrato `/v1/audio/speech` que D-05/D-06 querem).

### Pattern 2: Kani thin shim (~150 LoC) — derivar do servidor oficial
**What:** FastAPI expondo `/v1/audio/speech` + `/v1/audio/voices`. Carrega `KaniTTS('nineninesix/kani-tts-2-pt')` + `SpeakerEmbedder()` no startup.
**Source:** [CITED: github.com/nineninesix-ai/kani-tts-2-openai-server] — JÁ implementa o contrato.
**Example (loading + inference, do model card):**
```python
# Source: huggingface.co/nineninesix/kani-tts-2-pt
from kani_tts import KaniTTS, SpeakerEmbedder
model = KaniTTS('nineninesix/kani-tts-2-pt')          # ~3 GB VRAM
embedder = SpeakerEmbedder()                           # Orange/Speaker-wavLM-tbr, 128-dim
# default voice:
audio, _ = model("Texto em português")
# cloned voice:
spk = embedder.embed_audio_file("reference_voice.wav") # extrai do WAV de referência
audio, _ = model("Texto clonado", speaker_emb=spk)
model.save_audio(audio, "out.wav")                     # 22050 Hz, 16-bit mono WAV
```
Voice cloning no servidor oficial: voices são arquivos `.pt` (tensores de speaker embedding 128-dim) num diretório `speakers/`; `voice="speaker_1"` seleciona, `voice="random"` pula embedding. **Mapeamento p/ D-08:** ao criar `voice_id`, o shim extrai o embedding do WAV (via `embedder.embed_audio_file`), salva o `.pt` localmente E o WAV no MinIO; em `/v1/audio/speech` o `voice_id` resolve o `.pt`.

### Pattern 3: TTS reverse proxy (JSON→binary) — NÃO reutilizar audio.go
**What:** `proxy/audio.go` (STT) é multipart→JSON. TTS é JSON→binary audio (oposto). Precisa proxy próprio.
**Como espelhar:** Copiar a ESTRUTURA de `NewAudioProxy` (BuildDirector, Transport, ErrorHandler) mas:
- `ResponseHeaderTimeout` maior (síntese de texto longo pode levar segundos; derivar do RTF — 30s de áudio @RTF 0.2 ≈ 6s; usar ~30-60s timeout).
- **SEM** FlushInterval se non-streaming (default buffered, igual audio.go); SE suportar `stream_format:sse`, FlushInterval=-1 como o chat streaming.
- `ErrorHandler("tts", log)`.
**Example:**
```go
// proxy/tts.go — NEW (espelha audio.go:19-40 mas binary response)
func NewTTSProxy(upstreamURL string, log *slog.Logger, interceptors ...ProxyResponseInterceptor) (*httputil.ReverseProxy, error) {
    u, err := url.Parse(upstreamURL)
    // ... validar scheme/host (igual audio.go)
    return &httputil.ReverseProxy{
        Director:  BuildDirector(u),
        Transport: &http.Transport{ ResponseHeaderTimeout: 60 * time.Second, /* ... */ },
        ErrorHandler:   ErrorHandler("tts", log),
        ModifyResponse: ComposeInterceptors(interceptors...),
    }, nil
}
```

### Pattern 4: Static tier-0 embed (já existe a infra)
**What:** Embed já É um row tier-0 na seed (`0008_seed_upstreams.sql:10`: `local-embed, embed, 0, UPSTREAM_EMBED_URL`). Hoje o reconciler SOBRESCREVE esse slot via `OverrideTier0("embed", podURL)` quando o pod sobe.
**Mudança D-03:** Basta (a) remover `"embed"` do `newTier0OverrideMap()` (loader.go:87-93) e (b) remover as 3 chamadas `OverrideTier0("embed",...)`/`RestoreTier0("embed")` no primary reconciler (markReady:554, startDrain:507, closeLifecycle:620, recoverOpenLifecycle:1015). Resultado: `Resolve("embed",0)` sempre lê o row estático `UPSTREAM_EMBED_URL` (agora apontando para n8n-ia-vm). NENHUMA mudança de schema necessária para o embed em si.

### Anti-Patterns to Avoid
- **Reutilizar `proxy/audio.go` para TTS:** direção de dados oposta — quebra Content-Type/streaming. Proxy próprio (D-12).
- **Deixar embed no `OverrideTier0`:** o reconciler sobrescreveria o slot estático quando o pod sobe e o LIMPARIA quando o pod dorme (off-peak), quebrando RAG 24/7 — exatamente o bug que esta fase resolve.
- **TTS como static upstream:** o pod muda de IP a cada lifecycle Vast.ai → static URL ficaria stale. TTS PRECISA do override dinâmico (D-11).
- **Esquecer o `UNIQUE(role,tier)` + `CHECK(role IN ...)`** na tabela upstreams (0007:21-23): adicionar `tts` exige migration que relaxa o CHECK antes de seed.

## Common Pitfalls

### Pitfall 1: Kani-TTS-2-pt pode não ser PT-BR nativo (model card metadata)
**What goes wrong:** O model card `nineninesix/kani-tts-2-pt` lista language tags `en`/`es`/`ky` e diz "optimized primarily for English" na seção Limitations. O nome do repo é `-pt` e há um HF Space dedicado `KaniTTS2-pt`, e fontes secundárias (MarkTechPost) chamam de "Portuguese (PT) version".
**Why it happens:** O card foi templado do modelo EN (boilerplate copiado); a metadata `language:` não foi atualizada para o fine-tune PT.
**How to avoid:** **Wave 0 gate obrigatório** — antes de qualquer código de produção, baixar o modelo e gerar uma frase PT-BR ("Olá, seu boleto venceu ontem"), validar qualidade/pronúncia com o operator. Se a qualidade PT for ruim, opções: (a) `kani-tts-2-en` + texto PT (degradado), (b) fine-tune via `kani-tts-2-pretrain`, (c) reconsiderar engine. Ver Open Question 1.
**Warning signs:** Pronúncia anglicizada, prosódia errada.

### Pitfall 2: CUDA 12.8+ requirement do servidor oficial Kani vs base image llama.cpp b9191
**What goes wrong:** O servidor oficial Kani lista "CUDA 12.8+, Ampere+". A base do pod é `llama.cpp:server-cuda-b9191` (CUDA 12.x). NeMo NanoCodec + transformers podem exigir runtime CUDA específico.
**Why it happens:** Stack de TTS tem deps PyTorch/CUDA pesadas; o pod já tem 4 venvs Python (speaches, infinity). Kani adiciona um 5º com torch+nemo (vários GB).
**How to avoid:** Validar no Wave 0 que torch+nemo do Kani rodam GPU na imagem do pod (5090 = Ada/Blackwell, Ampere+OK). Medir tamanho da imagem (já é grande). Considerar venv dedicado `/opt/kani-venv`. Confirmar disk filter (40 GB pode ficar apertado — Kani venv + weights NanoCodec ~vários GB; rever Decision 4 da WAVE0-GATES se necessário).
**Warning signs:** `RuntimeError: CUDA capability`, imagem >X GB falha no pull dentro do cold-start budget.

### Pitfall 3: Pitfall #11 / tech debt #9 — emerg RestoreTier0("llm") apaga o slot do primary
**What goes wrong:** `emerg/reconciler.go:511` chama `RestoreTier0("llm")` no cutback (EmergencyActive→Recovering). Se o primary pod ESTÁ Ready (escreveu o slot llm via `markReady`), o emerg apaga esse override → tier-0 llm cai para o row estático `local-llm` (URL stale do último pod). O primary reconciler só escreve o slot em `markReady`/`recoverOpenLifecycle` (one-shot) — nunca re-afirma no Ready-tick. Com `tts` adicionado, o mesmo vetor afeta tts (e stt no futuro).
**Why it happens:** emerg e primary compartilham o MESMO mapa `tier0Override` mas o emerg só conhece "llm" e limpa unilateralmente.
**How to avoid (design do fix D-13):** Em `evaluateReady` (reconciler.go:394), quando FSM=Ready, **re-afirmar o override se o slot estiver vazio**. O Loader precisa de um getter `Tier0OverrideURL(role) (string,bool)` (NÃO existe hoje — adicionar). evaluateReady passa a:
```go
func (r *Reconciler) evaluateReady(ctx, now, log) {
    // ... drain check existente ...
    // Pitfall #11 fix: re-assert tier-0 if cleared by emerg cutback
    if urls := r.activePodURLs.Load(); urls != nil && r.deps.Loader != nil {
        for _, role := range []string{"llm","stt","tts"} {
            if _, set := r.deps.Loader.Tier0OverrideURL(role); !set {
                r.deps.Loader.OverrideTier0(role, stripPrimaryReadinessSuffix(roleURL(urls, role)))
                log.Warn("primary re-asserted tier-0 override (emerg cleared it)", "role", role)
            }
        }
    }
}
```
Roda a 1Hz (tick), então a janela de inconsistência é ≤1s. **Nota:** embed SAI deste loop (D-03) — só llm/stt/tts.
**Warning signs:** Após cutback do emerg, requests llm/tts caem em fallback OpenRouter/Piper mesmo com primary Ready.

### Pitfall 4: VRAM no 5090 com 4 modelos GPU
**What goes wrong:** Phase 6.6 DCGM mostrou ~24.7 GB usado / ~7.4 GB livre no 5090 (Qwen Q4_K_M ~17 + KV ~2 + whisper-large-v3 GPU ~3-4 + bge-m3 ~2). Embed SAI do pod (libera ~2 GB) e Kani ENTRA (~3 GB). Net: ~24.7 - 2 + 3 ≈ 25.7 GB usado / ~6.3 GB livre.
**Why it happens:** Kani é leve (~3 GB) mas soma com torch overhead do venv.
**How to avoid:** Net positivo de headroom porque embed libera ~2 GB. Confirmar em Live UAT (DCGM scrape). Headroom ~6 GB é confortável. Se OOM, fallback existente: whisper-medium / cap KV (`--n-keep`).
**Warning signs:** CUDA OOM no llama-server sob KV peak (já mitigado em 6.6).

### Pitfall 5: Piper não é OpenAI-compat (precisa adapter)
**What goes wrong:** voice-api Piper expõe `POST /tts` com `{text, voice}` → raw **ulaw 8kHz** (`audio/basic`) [VERIFIED: SSH read de /app/server.py em voice-api-piper-1]. NÃO é `/v1/audio/speech`, e o formato (ulaw 8kHz telefonia) ≠ WAV 22050 do Kani.
**Why it happens:** Piper foi feito para Asterisk (telefonia).
**How to avoid (D-07):** O upstream tier-1 `voice-api-piper` precisa de um adapter. Duas opções: (a) adapter no gateway que traduz `/v1/audio/speech` JSON → `POST /tts` e converte ulaw→wav/pcm no response; (b) adicionar uma rota `/v1/audio/speech` ao voice-api (Bun) que faz proxy interno pro Piper. **Recomendação:** opção (a) no gateway é mais isolada, mas como o degradado é "voz default" (sem clone), o response_format pode ficar ulaw/wav simples. O probe `tts` tier-1 deve bater num endpoint que retorne 200+audio. Validar com operator qual formato os clientes off-peak aceitam.
**Warning signs:** Cliente recebe ulaw 8kHz esperando WAV 22kHz → áudio quebrado off-peak.
