## DEVIATIONS (live-environment fixes vs the planned compose)

The CPU host is **Intel i7-8700 (Coffee Lake) — AVX2 only, NO AVX512**. Three fixes were required to get a stable service (all in `deploy/embed/docker-compose.yml`):
1. **`--dtype int8` → `float32`** then **`--engine optimum` → `torch`**: Infinity 0.0.77 always selects the HF repo's pre-quantized `model_qint8_avx512_vnni.onnx` (requires AVX512-VNNI) regardless of `--dtype`, crash-looping (200+ restarts, SIGILL) on this CPU. `--engine torch` loads via sentence-transformers/torch fp32, bypassing the ONNX variants → runs on AVX2.
2. **Healthcheck `/v1/health` → `/health`**: Infinity serves `/health` UNPREFIXED even with `--url-prefix /v1` (only `/v1/models` + `/v1/embeddings` get the prefix). Verified live: `/health`=200, `/v1/health`=404.
3. **`INFINITY_MODEL_WARMUP=true` → `false`** + `start_period` 180s→240s, retries 3→5: the warmup timing-benchmark runs all batch sizes and exceeds 180s on this fp32 CPU, blocking request serving past the healthcheck window → unhealthy. Disabled so the server is ready ~45s after load. Also added `cpus: 3.0` cap (commit `08acef1`) to protect the shared n8n host (load was ~15% / 6 cores, 11Gi free — measured).

Perf note: fp32 e5-large on this CPU = ~19.5 emb/s (batch 32); single query ~100-200ms. Acceptable for query-time RAG (1 embed/query). Heavy corpus re-embed (Phase 8) should use the GPU opportunistically.
