# Phase 06.7 Plan 05: Primary-Pod Chatterbox TTS Service Summary

Built the GPU TTS service on the primary pod — a thin FastAPI shim wrapping Chatterbox Multilingual (`ResembleAI/chatterbox`, MIT) serving the OpenAI `POST /v1/audio/speech` contract + `GET /health` on port 8003 as the 5th supervisord child, with zero-shot voice cloning that survives ephemeral pod replacement via a lazy MinIO reference-WAV fetch (no `.pt` embedding). The Infinity embed child left the pod (D-03).

## What Was Built

### Task 1 — `chatterbox_server.py` + unskipped tests (commit c1a932b)
- `POST /v1/audio/speech` accepting OpenAI fields (`model`, `input`, `voice`, `response_format`), returning binary WAV (RIFF, 24000 Hz 16-bit mono) or raw PCM16. `language_id` pinned server-side to `pt` (D-06).
- `GET /health`: 200 only when `model_ready` is True and the model object is loaded; 503 while loading at startup (REVIEWS action #4). FastAPI `lifespan` loads `ChatterboxMultilingualTTS.from_pretrained("cuda")` then runs a one-shot warmup synth so the first real request is not cold.
- Zero-shot voice resolution per the voice-persistence contract: `voice in (None,"","random")` -> `audio_prompt_path=None` (default voice); otherwise validate the `voice_id`, use a local cached WAV on hit, else lazily fetch `<CHATTERBOX_S3_VOICE_PREFIX>/<voice_id>.wav` from MinIO, cache it, and pass it as `audio_prompt_path`; no S3 object -> 404. There is NO `.pt` embedding anywhere — the reference WAV is the single durable artifact.
- Security: `voice_id` allowlist regex `^[A-Za-z0-9_\-\.]+$` + explicit `..` rejection (T-06.7-08); `input` capped to `CHATTERBOX_MAX_INPUT_CHARS` (T-06.7-09); `response_format` restricted to `{wav,pcm}`.
- Config is env-driven (`CHATTERBOX_MODEL_CACHE_DIR`, `CHATTERBOX_VOICE_CACHE_DIR`, `CHATTERBOX_S3_VOICE_PREFIX`, `CHATTERBOX_MAX_INPUT_CHARS`, `CHATTERBOX_LANGUAGE_ID`, MinIO creds), no hardcoded dirs/caps.
- Heavy imports (`torch`/`chatterbox`/`torchaudio`/`boto3`) are deferred into the model-load and S3 paths, so the module imports and unit-tests with NO GPU and NO S3.
- The 5 owned Plan 02 pytest stubs were unskipped and now assert real behavior (mocked model + MinIO), plus 4 added edge tests (404, input-cap, unsupported-format, pcm): **9 passed, 0 skipped**.

### Task 2 — Dockerfile Chatterbox stage + supervisord swap (commit dd82831)

## Adapt-vs-Shim Decision (D-05 / GATE 2)

**Chosen: thin FastAPI shim wrapping `chatterbox-tts` directly** (NOT packaging `devnen/Chatterbox-TTS-Server`).

Rationale: we need only the OpenAI `/v1/audio/speech` + `/health` + voice-resolution surface; the devnen server ships a Gradio UI, predefined-voice management, and config layers we would have to trim/pin. The shim is a single auditable file with our exact contract (server-pinned `language_id=pt`, env-driven paths/caps, zero-shot S3 WAV resolution, path-traversal guard) and stays importable + unit-testable without a GPU because the heavy imports are deferred. The served contract is identical either way.

## Image Size + Model-Cache Strategy (Task 2 acceptance — disk budget)

- **MEASURED final image size: NOT captured in this plan.** Docker is not installed on the control host (`ops-claude`); the primary-pod image builds in CI (`.github/workflows/build-primary-pod.yml`) and on a paid GPU host. Per the execution autonomy boundary, the full image build (cu124 torch ~1.6GB + Chatterbox HF weights ~3GB) is a heavy/paid operation and was NOT run autonomously.
- **Baseline estimate (GATE 2):** Chatterbox HF weights ~3.0GB + torch ~1.6GB + transformers 5.2.0/perth/s3tokenizer deps -> estimated **added layer ~6-7GB**. Fits the 60GB pod disk; RESEARCH Pitfall 2 warns 40GB can be tight stacked with llama.cpp + STT.
- **Model-cache strategy:** weights are NOT baked into the image — `HF_HOME`/`TORCH_HOME` point at `CHATTERBOX_MODEL_CACHE_DIR` (`/opt/chatterbox-data/models`) and Chatterbox downloads from HF on first load at pod onstart (cold load ~30-40s, warmed once at startup). This keeps the image layer to the Python/torch stack only (≈ the ~6-7GB estimate minus the ~3GB weights, so the baked layer is closer to ~3.5-4GB; weights land on the pod's ephemeral disk at runtime).
