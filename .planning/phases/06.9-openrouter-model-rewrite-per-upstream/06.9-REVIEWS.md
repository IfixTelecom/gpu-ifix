# Cross-AI Plan Review — Phase 06.9

## Gemini Review

# Phase 06.9: OpenRouter model-rewrite per-upstream — Plan Review

## 1. Summary
This plan bundle is of **exceptional quality**, providing a technically rigorous and architecturally sound solution to a "silent failure" bug that has persisted since Phase 03. By pivoting from request-edge model aliasing to a schema-driven, per-Director approach (Option B), the gateway gains the extensibility needed to support diverse tier-1 fallback targets without code-level churn. The plans exhibit a high degree of defensive engineering—notably the boot-time config fail-fast for URL suffixes and the implementation of a multipart-aware rewriter for Whisper that preserves binary integrity. The testing strategy is particularly strong, closing the "blind mock" gap that originally allowed this bug to go undetected.

## 2. Strengths
- **Schema-Driven Extensibility:** Widening the `model_aliases` PK to `(alias, upstream_name)` is the correct architectural choice, ensuring that a single alias (like `qwen`) can resolve to different slugs depending on whether it is handled locally or by a specific external provider.
- **Technical Rigor in Audio Handling:** The Whisper multipart rewriter (Plan 03, Task 2) correctly identifies the risk of binary corruption and specifies a stream-based `io.Copy` approach, avoiding common pitfalls with string conversions.
- **Defensive Engineering:** The boot-time fail-fast validation for URL suffixes (`/v1`) is a high-signal improvement that prevents common operator misconfigurations from reaching production.
- **Testing Blind-Spot Closure:** Extending `upstreamMock` to capture and assert on forwarded request bodies (Plan 05) directly addresses the root cause of why this bug was not caught by earlier integration suites.
- **Operator Surface Improvements:** The addition of `gatewayctl breaker` and `model-alias` subcommands provides the necessary tooling for deterministic UAT and runtime model management without requiring redeploys.

## 3. Concerns
- **UAT Blocking Prerequisite (MEDIUM):** Scenarios S2 (Whisper) and S3 (Embed) require a real OpenAI API key. If the operator does not have one available, these scenarios will fail, potentially stalling the phase closure. This is noted in Plan 06, but remains a logistical dependency.
- **Migration Down Uniqueness (LOW):** If an operator has manually added custom model aliases sharing the same name across different upstreams, the Down migration (Plan 01) will fail when attempting to restore the `PRIMARY KEY (alias)`. This is a known risk in schema widening, and the plan appropriately documents the manual cleanup required in such an edge case.
- **In-Process vs. Redis Breaker Force (LOW):** Plan 04 Task 2 implements `force-open` via Redis. While this mirrors the `shed-force` pattern, there is a minor latency window (up to the next breaker evaluation tick) before the force takes effect. For UAT purposes, this is acceptable.

## 4. Suggestions
- **Upstream Validation in CLI:** In `gatewayctl model-alias set`, consider validating that the provided `--upstream` exists in the `upstreams` table to prevent orphaned alias rows.
- **Detailed Error Logging in Directors:** If `models.RewriteJSONModel` or the multipart rewriter fails, consider logging the error at `DEBUG` or `WARN` level (excluding sensitive tenant data) to assist in future bisection of malformed client requests.
- **Audit Column Documentation:** As noted in Research O.Q. #4, adding a `COMMENT ON COLUMN` to `audit_log.upstream` that enumerates the new canonical names (`openrouter-chat`, etc.) would improve long-term schema discoverability for data analysts.

## 5. Risk Assessment: LOW
The risk is **LOW**. The implementation uses established patterns (JSON rewrap via `json.RawMessage`, composition over inheritance for Directors) and is backed by a multi-layered testing strategy including unit, integration (testcontainers), and live UAT. The fallback behavior (returning the alias unchanged on resolver miss) ensures that the gateway remains functional even if migrations or seeds are partially applied. The migration path for the primary data-class (tier-0) is backfilled and verified, ensuring no regression for local pod traffic.

**Verdict:** PROCEED. All 6 plans are ready for execution.

## Codex Review

## Summary

The plan bundle is strong on root-cause traceability and regression coverage: it correctly identifies that the rewrite must happen after upstream selection, adds schema support for per-upstream model targets, and closes the exact testing blind spot that let the OpenRouter 404 bug survive. The main concern is scope control. Plans 03 and 04 expand from “model rewrite fix” into multipart rewriting, breaker override semantics, gatewayctl CRUD, Redis read-path changes, live UAT orchestration, and cascade documentation. That can still work, but Plan 04 in particular risks becoming a second feature project inside the fix phase.

## Strengths

- Clear causal chain from bug evidence to implementation: dispatcher selects `openrouter-chat`, but body still has `model:"qwen"`.
- Correct architectural pivot: per-upstream resolution via `(alias, upstream_name)` is the right long-term shape.
- Good test design improvement: body-capturing upstream mocks directly prevent the previous false-positive integration tests.
- Strong coverage map across unit, integration, migration round-trip, and live UAT.
- Good handling of OpenRouter URL gotcha by making `/api` vs `/api/v1` fail fast.
- Whisper multipart rewrite is explicitly tested for byte-identical audio preservation, which is the right edge case.
- Migration plan is mostly conservative: additive column, backfill, seeded tier-1 rows, composite PK.
- Live UAT scenarios are well aligned with the actual phase closure criteria.

## Concerns

- **HIGH: Plan 04 breaker force-open likely exceeds phase scope.** Adding `gatewayctl breaker force-open`, Redis force override keys, and breaker read-path changes is operationally useful, but it is not necessary to fix model rewriting. It introduces new runtime behavior and needs its own correctness/security review.

- **HIGH: Breaker override surface is security-sensitive.** A CLI that can force upstreams open/closed can intentionally route traffic to paid external providers or disable fallback. The plans do not sufficiently specify authorization boundaries, audit logging, TTL enforcement, or deployment access assumptions.

- **HIGH: Migration Down remains risky/inconsistent.** Plan 01 says Down preserves `upstream_name` but restores PK on `(alias)` after deleting only seeded tier-1 rows. That is safe only if no operator-created extra rows with duplicate aliases remain. If `gatewayctl model-alias set` can insert new rows, Down may fail on duplicate aliases.

- **MEDIUM: Plan 03 may miss local-tier rewrite symmetry.** The narrative says per-upstream rewrite should happen in directors, but the concrete director tasks only cover OpenRouter, OpenAI Whisper, and OpenAI Embed. If local proxies still need alias resolution after removing `models.Handler`, local LLM/embed behavior must be verified explicitly.

- **MEDIUM: OpenRouter director rewrite may allocate/re-marshal multiple times despite “single pass” claims.** Calling `models.RewriteJSONModel`, then `injectProviderOrder`, then `injectStreamOptionsIncludeUsage` is three JSON passes unless refactored into one helper. That is acceptable, but the plan should not claim single-pass unless it actually implements it.

- **MEDIUM: Whisper multipart helper has validation ambiguity.** The plan says validate `newModel` against a regex, but also says resolver miss passes alias unchanged. If a client supplies a weird model alias and resolver misses, the regex may reject and silently forward original body. That behavior should be explicit and tested.

- **MEDIUM: Plan 04 model-alias CLI uses ad-hoc SQL while sqlc is the established pattern.** The rationale avoids regen churn, but this creates a second data-access style for a core table. If included, it should at least share validation and tests with resolver expectations.

- **MEDIUM: Live UAT depends on OpenAI credentials not yet present.** The plan marks this, but S2/S3 cannot close without it. This should be a hard precondition before execution, not a late operator checklist item.

- **MEDIUM: Integration tests may be hard to wire if `freshSchema` only returns Redis and not Postgres pool.** The plan assumes a resolver can be instantiated from the integration helper. Execution should verify helper signatures early.

- **LOW: Documentation and cascade closure are over-specified.** Three separate commits and detailed verification edits are fine, but they should not block code correctness if live UAT evidence is already captured.

- **LOW: `UPSTREAM_LLM_OPENROUTER_MODEL` treatment is inconsistent across docs.** Some sections say deprecated/no-op, others mention retained fallback override. Final plans mostly choose no-op plus CLI override, but all docs should converge.

## Suggestions

- Split Plan 04 into two parts:
  - Required for this phase: config `/v1` fail-fast.
  - Optional/follow-up: `gatewayctl breaker` and `gatewayctl model-alias`.
- If breaker forcing stays in scope, require:
  - TTL mandatory with sane max.
  - Audit log entry for force-open/force-close.
  - Explicit Redis key namespace and runtime read path.
  - Operator-only deployment note.
  - Tests proving TTL expiry restores normal breaker behavior.
- Add a migration Down guard:
  - Before restoring `PRIMARY KEY(alias)`, delete or reject all non-tier0 duplicate alias rows.
  - Or document Down as not supported after operator-created per-upstream aliases.
- Add explicit local upstream director coverage:
  - `local-llm`: `qwen -> qwen`.
  - `local-embed`: `bge-m3 -> bge-m3` or whatever local target is expected.
  - Confirm removing `models.Handler` does not regress tier-0.
- Make OpenRouter rewrite implementation honest:
  - Either refactor to one JSON helper that rewrites model, provider, and stream options together.
  - Or keep three passes and remove “single rewrap pass” from acceptance criteria.
- Add tests for malformed JSON/multipart:
  - JSON invalid body forwards unchanged.
  - Multipart missing `model` forwards unchanged or inserts target, whichever is intended.
  - Multipart duplicate `model` fields behavior is defined.
- For `gatewayctl model-alias set`, validate `alias`, `upstream`, and `target` formats and reject whitespace/control characters.
- Ensure no secrets appear in UAT docs:
  - Commands should reference env vars, never paste real tokens.
  - Evidence boxes should instruct operators to redact auth headers and keys.
- Make OpenAI key availability a Wave 5 entry gate:
  - Do not start live UAT until direct OpenAI Whisper and embedding calls have been validated.
- Add one integration test for the exact original failure:
  - Primary breaker open.
  - Gateway receives `model:"qwen"`.
  - Fake OpenRouter rejects `model:"qwen"` but accepts `qwen/qwen3.5-27b`.
  - This proves the fix would have caught the historical bug.

## Risk Assessment

**Overall risk: MEDIUM.**

The core fix design is sound and likely closes the OpenRouter model-rewrite bug and Phase 03 SC-1 once implemented. The schema-driven resolver plus director-level rewrite is the right dependency order, and the proposed integration tests directly target the previous blind spot. Risk rises because the bundle expands into operational controls and CLI behavior that are not strictly necessary for the bug fix. The highest-risk areas are the breaker force-open runtime path, migration Down behavior after new operator-created aliases, and ensuring removal of `models.Handler` does not accidentally regress local tier-0 routing.

## Consensus Summary

Two independent reviewers (Gemini, Codex) both rate the bundle as technically sound on the root-cause fix path (Option B schema-driven per-upstream rewrite) with overlapping concerns around scope control and live-UAT robustness.

### Agreed Strengths

- Root-cause traceability — both reviewers note plans correctly localize the rewrite at the post-resolution Director boundary, not at the request edge.
- Test gap closure — body-capturing upstreamMock + 3 model-rewrite assertions directly retire the "blind mock" that let the OpenRouter 404 bug survive Phases 04–08.
- Migration 0026 PK widening to (alias, upstream_name) is recognized as the right schema shape for multi-tier-1 extensibility.
- Boot-time fail-fast on UPSTREAM_*_URL ending in /v1 prevents silent path-doubling regressions.

### Agreed Concerns (HIGH priority for replan)

- **Scope creep in Plan 04** — Both reviewers flag that operator surface (gatewayctl breaker {force-open,force-close} + model-alias CRUD) is a meaningful feature project bolted onto a fix phase. Codex calls it explicit; Gemini implies via "defensive engineering" framing. Recommendation: consider splitting Plan 04 into 04a (config fail-fast — required for fix) + 04b (gatewayctl CLI — promote to its own phase, or gate behind explicit success of 04a).
- **Live UAT blast radius (Plan 06)** — Both reviewers want sharper guardrails on the operator-driven S1–S6 sequence: explicit OpenRouter spend cap, breaker restore on abort, rollback procedure if migration 0026 needs to revert mid-UAT.
- **Multipart rewriter (Plan 03 Whisper)** — Binary-integrity preservation is praised as a strength but both flag it as the highest-risk single piece of new code in the bundle; merits dedicated integration test beyond the body-capture mock.

### Divergent Views (worth investigating)

- **Migration 0026 Down safety** — Gemini does not flag data loss risk on Down; Codex implicitly trusts the round-trip test in Plan 05. Neither explicitly verifies that Down preserves the 3 tier-1 seed rows or documents what data the operator should expect to lose. Worth a 1-line note in 06.9-01-PLAN.md confirming Down behavior on production-shaped data.
- **Phase ordering** — Gemini treats the 6-wave sequencing as well-ordered; Codex hints that some Plan 04 work could move earlier or be deferred entirely.

### Recommendation

Proceed to execution with two replan touch-ups:
1. Add explicit Plan 06 guardrails (spend cap, breaker-restore, migration-rollback) — low-cost insertion.
2. Decide on Plan 04 split. If keeping bundled, document explicitly why gatewayctl model-alias CRUD must ship in the fix phase (live-UAT operator surface dependency) — otherwise defer 04b to a follow-up phase.
