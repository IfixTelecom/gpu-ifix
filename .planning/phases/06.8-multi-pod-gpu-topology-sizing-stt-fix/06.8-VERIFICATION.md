### Root cause (Task 1 — Diagnosis)

The diagnosis (commit `37a91cb`, 06.8-ALLOWLIST-DIAGNOSIS.md) DISPROVED the three hypotheses the gap-closure plan asked to test:

- **H1 (wire encoding):** the marshaled JSON the reconciler sends (`{"machine_id":{"in":[43803]},…}`) is byte-equivalent to the A2 hand-built query that returned 1 offer for machine 43803 earlier in the day. Vast `in` operator works (symmetric to `notin` proven by 50-offer probe with 0 hits in the blocklist).
- **H2 (FilterBelowCap eliminates):** dph 0.401 ≤ cap 0.60 + epsilon — passes trivially. Even with surcharge variance, 0.20 of headroom is well outside any plausible surcharge.
- **H3 (env empty at runtime):** the env was set in `.env` and the compose passthrough was added (UAT-03 docs); the binary just didn't have a consumer for the env var.

**Actual root cause (H4):** **deploy staleness.** The running `ai-gateway-dev` container was on a `:latest-dev` image built `2026-05-21T21:13:54Z` — ~2h20m BEFORE commit `6f57698` (the allowlist feature, `2026-05-21T23:33Z`). `strings | grep` on the deployed binary returned **0 hits** for `PRIMARY_VAST_MACHINE_ALLOWLIST` / `WithMachineAllowlist` / `PrimaryVastMachineAllowlist` / `primary allowlist exhausted`. The Portainer webhook fired but the local Docker daemon retained the previous digest (the documented STATE.md 2026-05-19 redeploy gotcha). Source on `develop` was correct.

Post-force-pull `:latest-dev`: all 4 symbols present, count=1 each. The "fix" was a `docker compose up -d --force-recreate gateway` on `vps-ifix-vm`, not a source edit. `git diff --quiet -- gateway/` clean at diagnosis time.
