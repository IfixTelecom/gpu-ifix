## TL;DR — Root cause is a DEPLOYMENT STALENESS gap, not a Go source bug

The Go source on `develop` (commit `6f57698` `feat(gateway): add PRIMARY_VAST_MACHINE_ALLOWLIST + PRIMARY_NUM_GPUS for multi-pod topology`, 2026-05-21T23:33Z) is correct. **But the deployed `ai-gateway-dev` container is running an OLD `:latest-dev` image** that was built at `2026-05-21T21:13:54Z` — **2h 20m BEFORE** the allowlist commit landed. The deployed binary literally does not contain the allowlist code path. The runtime evidence operators saw at Plan 06.8-03 UAT (`gateway picked 76546 / 47525 instead of allowlisted 43803`) is the natural consequence: the gateway binary has only the broaden pass, picks global-cheapest qualifying offer, ignores `PRIMARY_VAST_MACHINE_ALLOWLIST` env entirely.

**Fix (proposed):** force-pull the current `:latest-dev` (which contains commit `6f57698` + later) on `vps-ifix-vm` and recreate the gateway container. Already done in passing during diagnosis (the registry's `:latest-dev` digest is now `sha256:0735af41cf39e1c86b477b68247db780e9915f6380c6b4f7a762161ee6cda112` — pulled to local; container recreate NOT yet performed, deferred to the operator-approved Task 2/checkpoint flow). No `gateway/*.go` change required for behavior.

Optional Task 2 (operator decision at the checkpoint): add a defensive regression test that pins the WithMachineAllowlist + reconciler-pass composition in CI, so a future image build can never silently strip the symbols. This is hardening — does not change runtime behavior.

## 4. Why the natural hypotheses didn't explain it

- **H1 (wire encoding):** the marshaled JSON is byte-identical to the proven-working A2 hand-built query. Vast `in` operator works (proven by the A2 1-offer return earlier today + a fresh probe with `notin` excluding blocked machines from 50 results).
- **H2 (FilterBelowCap eliminates):** dph 0.401 ≤ cap 0.60 + epsilon — passes trivially.
- **H3 (env empty at runtime):** the env was set in `.env` and the compose passthrough was added. The current container env confirms `PRIMARY_VAST_MACHINE_BLOCKLIST=55942,45778` and other PRIMARY_* vars are populated, so the env passthrough mechanism works. The flaw was deeper: **the binary that consumes the env doesn't have the `PrimaryVastMachineAllowlist` field at all** — even if the env had been set, `parseInt64CSV` is invoked by code that was never compiled in.

H4 (the actual cause): **deployment staleness — registry has the right image, local Docker daemon kept the previous digest after recreate.** The compose / Portainer webhook fired (per CI logs run `26271486310`) but the local image was not refreshed.

## 5. Responsible source line: NONE — but the test-coverage gap is real

There is no `gateway/*.go` line to repair. The source is correct.

The systemic gap is that **there is no test that proves the running binary contains the allowlist code path** before a UAT run goes live. The existing `TestWithMachineAllowlist` (types_test.go L114) covers the helper in isolation; the existing `TestDefaultSearchFilter_NumGPUs` (L97) covers num_gpus knob. Neither test would have caught the stale-image scenario. A defensive measure is a regression test at the reconciler level that asserts the allowlist-first pass restricts pickable[0] to the listed machine_id when an in-list offer exists alongside a cheaper out-of-list offer — this would still pass on the buggy binary if it had been built from current source, so it doesn't detect THIS deployment-staleness root cause. But it does pin the contract for the next refactor.
