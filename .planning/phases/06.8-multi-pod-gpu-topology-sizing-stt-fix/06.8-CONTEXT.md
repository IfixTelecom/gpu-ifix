# Phase 06.8: Multi-pod GPU topology + sizing + STT fix - Context

**Gathered:** 2026-05-21 (seeded from the 2×3090 live spike — see memory `vast-multi-gpu-cdi-risk`)
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the gateway able to run the primary pod across **multiple GPU topologies** (not just a single 5090), choosing on cost while staying resilient to scarce/interruptible Vast inventory, and **fix the STT model-resolution bug** that blocks `/v1/audio/transcriptions` on every topology.

The 2026-05-21 spike validated a **single pod on a 2×RTX 3090 machine** (48GB total) as the new cost/availability sweet spot:
- ~$0.29–0.40/h ($212–290/mo) vs single 5090 (~$0.74/h viable, $540/mo) — ~60% cheaper.
- 3090×1/×2 inventory is **deeper and price-stable** (9–10 offers) than cheap-5090 (1 offer then a cliff to ~$0.94/h).
- **Keeps the gateway single-primary architecture unchanged** (one pod, one lifecycle).

Net result after this phase:
- The primary reconciler can provision + health-poll a **2×3090 single-pod** (via `PRIMARY_NUM_GPUS=2`), preferring known-good hosts (`PRIMARY_VAST_MACHINE_ALLOWLIST`).
- Documented + tested **fallback topology ladder** for when the preferred shape is unavailable (Vast is spot — no reservation): 2×3090 → single 5090 → … with emergency/tier-1 covering gaps.
- **STT works** end-to-end on the chosen topology (whisper model resolves + loads on GPU).

**Not this phase:**
- SEED-002 emergency = full primary-pod hot-standby + embed-on-GPU tiering → **next phase** (06.8 decides the GPU the emerg hot-standby will mirror, so it runs first).
- True dual-primary (2 separate pods, role-split LLM↔STT/TTS) routing in the gateway → deferred (single-pod 2×3090 avoids it).
- Switching LLM/STT/TTS engines or models (Qwen3.6 + whisper-large-v3 + Chatterbox stay).

</domain>

<decisions>
## Implementation Decisions

### Gateway knobs — ALREADY LANDED (commit 6f57698, this session)
- `PRIMARY_NUM_GPUS` (default 1) → `num_gpus:{eq:N}` in `vast.DefaultSearchFilter`. Set 2 for 2×3090.
- `PRIMARY_VAST_MACHINE_ALLOWLIST` → preference pass: reconciler searches allowlist-only first (`machine_id:{in}`), then broadens to the full qualified search when unavailable (Vast spot marketplace — hard filter would block provisioning). Mirrors `PRIMARY_VAST_MACHINE_BLOCKLIST` on the success side.
- Both default to no-op (numGPUs=1, allowlist empty). Unit tests added. **Validate these actually drive a real 2-GPU provision on a live pod in this phase.**

### 2×3090 single-pod — validated facts (spike, machine_id 43803)
- **CDI multi-GPU works** on this host: both 3090s visible in-container (`nvidia-smi` idx 0+1, 24GB each). The blocklisted machines (36773,53128,38389,51466) are specific broken hosts, NOT all multi-GPU. **Validate CDI per-host; add good ones to allowlist.**
- **llama.cpp auto-tensor-splits Qwen** across both GPUs with `-ngl 99` (~8.5GB GPU0 + ~9.4GB GPU1 = ~18GB). **No `CUDA_VISIBLE_DEVICES` pinning needed** — 48GB gives huge headroom (full stack ~27GB: LLM 18 + TTS 3.6 + whisper ~3 + embed ~2).
- **Ampere (sm_86) runs the cu128 image fine** (cu128 is backward-compatible). LLM ~42 tok/s on the split (vs ~77 single-5090 — inter-GPU PCIe overhead is the cost; acceptable for chat).
- **GHCR private pull needs NO image_login** in the Vast create — account-side docker creds apply.
- Vast bills **per-second** (not full-hour); boot+download+idle-running all bill; destroy stops compute billing.

### STT fix (folded in per operator) — root cause confirmed, MUST validate on pod
- Symptom: POST `/v1/audio/transcriptions` → 500 `huggingface_hub.errors.CacheNotFound: /root/.cache/huggingface/hub ... set HF_HUB_CACHE`.
- Root cause: the whisper weights tarball is **flat** (`model.bin`+`config.json` raw, `tar -C WHISPER_DIR .` in `pod/scripts/upload-weights.sh:80-91`) extracted to `/weights/whisper`, but **speaches 0.9.0rc3 resolves the model via the HF hub cache** (`models--Systran--faster-whisper-large-v3/snapshots/<hash>/`), not a flat dir.
- Candidate fixes (planner picks + validates on a live pod — do NOT blind-edit, per CLAUDE.md debugger rule):
  - (i) reshape flat→HF-hub-cache layout in the onstart + set `HF_HUB_CACHE` in `[program:speaches]` env (supervisord.conf);
  - (ii) point speaches at a local model path (if 0.9.0rc3 supports it);
  - (iii) make `upload-weights.sh` produce an HF-hub-layout tarball (mirror however BGE works) + set `HF_HUB_CACHE`.
- This is image-level + affects ALL topologies. It also blocked S6/STT in 06.7 UAT.

### Resilience posture (operator-confirmed)
- **Interruptible pods are acceptable** because emergency/fallback is effective: a preempted pod errors/timeouts → gateway calls the role's fallback. Do not pay the on-demand premium for "reservation"; Vast cannot reserve anyway.
- Allowlist is a **preference, not a guarantee**.

### Claude's Discretion (planner/researcher derive)
- Whether to add a gateway live-UAT plan that actually force-ups a 2×3090 (`PRIMARY_NUM_GPUS=2`) and asserts the 4-endpoint health passes with 2 GPUs + port mapping works.
- Exact STT fix mechanism (research speaches 0.9.0rc3 source + iterate on a pod).
- Whether `num_gpus` should be `eq` (exact) vs `gte` (at least) — spike used `eq`.
- Optional: allowlist auto-add (gateway records machine_id after a successful provision) — currently manual env only.
- Fallback ladder encoding: config-driven topology list vs documented runbook.

- `vast.DefaultSearchFilter` numGPUs param + `WithMachineAllowlist` (just added).
- The `-p PORT:PORT` env-key port-publish mechanism (required — without it the Vast instance has no port mapping; the spike's first instance was unreachable).

- Manual launch reaches the pod via Vast `public_ipaddr` + the mapped host port for container 22 (sshd from `POD_DEBUG_SSH_PUBLIC_KEY`).

- `reconciler.provisionLifecycle` — already wired for `PrimaryNumGPUs` + allowlist (landed); needs live validation.
