### Scenario 1 — Force-provision + 4 services GREEN (Wave 0 supervisord 4-child-process model)

**Validates:** Strategy α custom multi-stage image builds + supervisord
co-locates 4 services + Vast.ai exposes 4 ports correctly + FSM Asleep →
Provisioning → Ready transition + gateway-side OverrideTier0 for 3 roles.

**Pre-conditions:** FSM at ASLEEP, no live lifecycle.

**Steps:**

```bash
ssh vps-ifix-vm

T0=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
echo "T0=$T0"

# Trigger force-provision (works under DISABLED=true per reviews #2)
docker exec ai-gateway-dev_gateway /gatewayctl primary force-up --reason "uat_scenario_1_wave0_supervisord"

# Poll FSM state every 60s until state=ready (cold-start budget = 40min)
for i in {1..40}; do
  echo "=== iteration $i / $(date) ==="
  docker exec ai-gateway-dev_gateway /gatewayctl primary state --format=json
  sleep 60
done

# Once state shows {"state":"ready", "pod_url": "http://..."}, capture pod
# IP + 4 host port mappings via the lifecycle row:
docker exec ai-gateway-dev_gateway /gatewayctl primary lifecycles --since 1h --format=json | jq '.[0]'

# Extract POD_IP + per-role host ports from the lifecycle's events JSONB
INST=$(docker exec ai-gateway-dev_gateway psql "$AI_GATEWAY_PG_DSN" -t -c "SELECT vast_instance_id FROM ai_gateway.primary_lifecycles WHERE ended_at IS NULL;" | xargs)
echo "INST=$INST"

# Query Vast.ai for the pod's public IP + port mappings
POD_INFO=$(curl -s -H "Authorization: Bearer $VAST_AI_API_KEY" "https://console.vast.ai/api/v0/instances/$INST/")
POD_IP=$(echo "$POD_INFO" | jq -r '.instances.public_ipaddr')
PORT_8000=$(echo "$POD_INFO" | jq -r '.instances.ports."8000/tcp"[0].HostPort')  # LLM
PORT_8001=$(echo "$POD_INFO" | jq -r '.instances.ports."8001/tcp"[0].HostPort')  # STT
PORT_8002=$(echo "$POD_INFO" | jq -r '.instances.ports."8002/tcp"[0].HostPort')  # Embed
PORT_9400=$(echo "$POD_INFO" | jq -r '.instances.ports."9400/tcp"[0].HostPort')  # DCGM
echo "POD_IP=$POD_IP"
echo "LLM=$PORT_8000  STT=$PORT_8001  Embed=$PORT_8002  DCGM=$PORT_9400"

# ASSERTION 1: all 4 endpoints respond from SAME pod IP (different host ports)
# This proves the supervisord 4-service single-container model.
curl -sS "http://$POD_IP:$PORT_8000/v1/models"  | jq -r '.data[0].id'  # Expect: qwen3.6-27b... or "qwen"
curl -sS "http://$POD_IP:$PORT_8001/health"     | head -10              # Speaches /health
curl -sS "http://$POD_IP:$PORT_8002/health"     | head -10              # Infinity /health
curl -sS "http://$POD_IP:$PORT_9400/metrics" | head -20                 # DCGM Prometheus metrics

# ASSERTION 2 (Wave 0 NEW): if POD_DEBUG_SSH_PUBLIC_KEY is set in pod env,
# SSH in and verify supervisorctl shows 4 RUNNING children.
ssh root@"$POD_IP" 'supervisorctl status'
# Expected output (4 lines):
#   llama       RUNNING   pid 12, uptime 0:25:00
#   speaches    RUNNING   pid 14, uptime 0:24:58
#   infinity    RUNNING   pid 16, uptime 0:24:56
#   dcgm        RUNNING   pid 18, uptime 0:24:54
```

**Expected:**

- FSM transitions `asleep → provisioning → ready` within ≤ 40min
  (Wave 0 Decision 6 cold-start budget).
- `gatewayctl primary state` shows `state=ready` with non-empty `pod_url`,
  `pod_instance_id`, `lifecycle_id`.
- All 4 endpoints respond from the same `POD_IP` on different host ports —
  proves supervisord 4-service single-container model.
- (If `POD_DEBUG_SSH_PUBLIC_KEY` set) `supervisorctl status` shows 4 children
  in `RUNNING` state: `llama`, `speaches`, `infinity`, `dcgm`.
- Gateway logs show `loader.OverrideTier0("llm", <url>)`,
  `loader.OverrideTier0("stt", <url>)`, `loader.OverrideTier0("embed", <url>)`.
- DB lifecycle row has `vast_instance_id IS NOT NULL`, `accepted_dph ≤ 0.4001`,
  `trigger_reason='manual_force_up'`, and the `events` JSONB contains
  `offer_accepted` + `healthy` entries.

**Pass/Fail criteria:**

- PASS: all 4 endpoints respond with valid bodies + FSM=ready + lifecycle
  row complete.
- FAIL: any endpoint returns 5xx/timeout, OR supervisorctl shows < 4
  RUNNING (a child crashed), OR FSM stuck in `provisioning` > 40min.

**Cleanup:** leave the pod live for Scenarios 2-4 (they reuse it).

**Cost estimate:** R$1-2 (~25-40min @ ~$0.40/h cap).

### Scenario 5 — Schedule out-of-window drain (EXPLICIT log assertions per BLOCKER 2)

**Validates:** `evaluateReady → startDrain("schedule_window_exited") →
evaluateDraining → Transition("drain_complete") → evaluateDestroying →
Transition("destroy_complete") → StateAsleep` end-to-end with **3 explicit
assertions** that the FSM transitions actually fired (not just that the pod
disappeared).

**Pre-conditions:** Scenario 1 PASS — primary pod live, FSM=ready.

**Steps:**

```bash
ssh vps-ifix-vm

# 1. Capture current hour in America/Sao_Paulo
NOW_HOUR=$(TZ=America/Sao_Paulo date +%H)
echo "NOW_HOUR=$NOW_HOUR"

# 2. Compute a DOWN_HOUR that simulates "we just left the peak window":
#    set DOWN_HOUR = NOW_HOUR (so IsInPeak() returns false on the next tick)
DOWN_HOUR=$NOW_HOUR
UP_HOUR=$((NOW_HOUR + 1))
[ $UP_HOUR -ge 24 ] && UP_HOUR=$((UP_HOUR - 24))
echo "Setting PRIMARY_POD_SCHEDULE_DOWN_HOUR=$DOWN_HOUR (current hour, simulating window just exited)"
echo "Setting PRIMARY_POD_SCHEDULE_UP_HOUR=$UP_HOUR (so next-up is 1h in the future)"

# 3. Update Portainer stack env vars via UI:
#    - PRIMARY_POD_SCHEDULE_DOWN_HOUR=$DOWN_HOUR
#    - PRIMARY_POD_SCHEDULE_UP_HOUR=$UP_HOUR
#    - PRIMARY_POD_SCHEDULE_DISABLED=false  (so the schedule tick runs)
#    Hit "Update the stack" → wait for the gateway container to recreate
#    (Portainer should re-pull the same image but with new env).
#
#    Or via Portainer API:
#    curl -X POST https://portainer3.ifixtelecom.com.br/api/stacks/<id>/update ...

echo "--- TRESTART TIMESTAMP ---"
T_RESTART=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
echo "T_RESTART=$T_RESTART"

# 4. Wait up to 90s for the gateway container to come back + emit the drain
#    transition. The reconciler.evaluateReady() will fire on the next
#    schedule tick (1Hz) and observe rule.IsInPeak()==false, then call
#    startDrain("schedule_window_exited").
sleep 90

# ============================================================
# ASSERTION 1 (BLOCKER 2 — Ready→Draining): explicit log/event grep
# ============================================================
#
# The reconciler does NOT emit a textual "transition ready→draining
# reason=schedule_window_exited" log line directly. Instead, observable
# surfaces are:
#   (a) Redis Pub/Sub channel gw:primary:events — typed events with
#       Type="transition" + State="draining" + Reason="schedule_window_exited"
#       AND Type="draining_started" + Reason="schedule_window_exited"
#   (b) The DB lifecycle row's events JSONB column contains
#       {"type": "draining_started", "reason": "schedule_window_exited", ...}
#   (c) gatewayctl primary state Hash mirrors state=draining
#
# The OPERATIVE ASSERTION uses surface (b) — the audit trail — because it
# survives Redis restarts and is the canonical record:
EVENTS_JSON=$(docker exec ai-gateway-dev_gateway /gatewayctl primary lifecycles --since 30m --format=json \
  | jq '.[0].Events')
echo "$EVENTS_JSON" | jq -e 'any(.[]; .type == "draining_started" and .reason == "schedule_window_exited")' \
  && echo "ASSERTION 1 PASS: draining_started event with reason=schedule_window_exited present" \
  || echo "ASSERTION 1 FAIL: no draining_started event with schedule_window_exited reason — drain logic did NOT fire within 90s"

# Cross-check via gateway state mirror
docker exec ai-gateway-dev_gateway /gatewayctl primary state --format=json
# Expected: {"state":"draining", ...}

# ============================================================
# ASSERTION 2 (BLOCKER 2 — Draining→Destroying): wait for grace
# ============================================================
#
# PRIMARY_POD_SCHEDULE_GRACE_RAMP_DOWN_SECONDS default = 300s (5min).
# evaluateDraining will Transition(Draining → Destroying) when
# inflight==0 OR grace elapsed. We assume no concurrent requests during
# the UAT, so inflight==0 should fire immediately on the next reconciler
# tick — but we wait the full grace_seconds + buffer to be safe.
echo "Waiting up to 360s for drain completion (grace=300s + 60s buffer)..."
sleep 360

EVENTS_JSON=$(docker exec ai-gateway-dev_gateway /gatewayctl primary lifecycles --since 1h --format=json \
  | jq '.[0].Events')
echo "$EVENTS_JSON" | jq -e 'any(.[]; .type == "drain_complete" or (.reason // "" | contains("drain_complete")))' \
  && echo "ASSERTION 2 PASS: drain_complete event present" \
  || echo "ASSERTION 2 FAIL: no drain_complete event — Draining→Destroying transition did NOT fire"

# Also check container log for the structured log line
docker logs ai-gateway-dev_gateway --since 10m 2>&1 | grep "primary drain complete; transitioning to Destroying"
# Expected: at least 1 match — slog.Info with inflight + elapsed_seconds + grace_seconds attrs.

# ============================================================
# ASSERTION 3 (BLOCKER 2 — FSM reaches StateAsleep terminal): final state
# ============================================================
#
# evaluateDestroying calls vastutil.BestEffortDestroy + closes lifecycle +
# Transition(Destroying → Asleep, "destroy_complete"). The lifecycle row
# should now have ended_at populated + shutdown_reason='schedule_drain'.
sleep 30
```

### Scenario 6 — Pitfall #11 emerg coexistence (per WARNING 5 pre-flight)

**Validates:** when both emerg and primary reconcilers want a pod
simultaneously, primary takes over and emerg force-destroys within 30s
(Pitfall #11 Option B locked per RESEARCH.md).

**Pre-conditions:**

- FSM at ASLEEP, no live primary lifecycle.
- emerg FSM at HEALTHY, no live emerg lifecycle.
- Prerequisites pre-flight check (d) confirmed `gatewayctl emerg
  force-provision` exists — otherwise this scenario cannot run.

**Steps:**

```bash
ssh vps-ifix-vm

# 1. Confirm starting state: both FSMs idle
docker exec ai-gateway-dev_gateway /gatewayctl emerg state --format=json
docker exec ai-gateway-dev_gateway /gatewayctl primary state --format=json

# 2. Trigger emerg force-provision (simulates Phase 6 reactive path)
docker exec ai-gateway-dev_gateway /gatewayctl emerg force-provision --reason "uat_pitfall11_test_emerg_first"

# 3. Wait for emerg to reach EMERGENCY_ACTIVE (cold-start ≤ 10min per
#    Phase 6 SC-1; could be longer if Vast inventory contention).
for i in {1..30}; do
  echo "=== emerg iteration $i / $(date) ==="
  EMERG_STATE=$(docker exec ai-gateway-dev_gateway /gatewayctl emerg state --format=json | jq -r '.state // "healthy"')
  echo "emerg state: $EMERG_STATE"
  [ "$EMERG_STATE" = "emergency_active" ] && break
  sleep 60
done

# Confirm emerg pod is up
docker exec ai-gateway-dev_gateway /gatewayctl emerg state --format=json
EMERG_INST=$(docker exec ai-gateway-dev_gateway psql "$AI_GATEWAY_PG_DSN" -t -c \
  "SELECT vast_instance_id FROM ai_gateway.emergency_lifecycles WHERE ended_at IS NULL;" | xargs)
echo "EMERG_INST=$EMERG_INST"

# 4. NOW trigger primary force-up — primary will provision a SEPARATE Vast pod
docker exec ai-gateway-dev_gateway /gatewayctl primary force-up --reason "uat_pitfall11_test_primary_takeover"

# 5. Wait for primary to reach state=ready (cold-start ≤ 40min)
for i in {1..40}; do
  echo "=== primary iteration $i / $(date) ==="
  PRIMARY_STATE=$(docker exec ai-gateway-dev_gateway /gatewayctl primary state --format=json | jq -r '.state // "asleep"')
  echo "primary state: $PRIMARY_STATE"
  [ "$PRIMARY_STATE" = "ready" ] && break
  sleep 60
done

# ============================================================
# ASSERTION: when primary marks Ready, emerg subscriber fires
# force-destroy on the emerg lifecycle within 30s
# ============================================================
#
# At the moment primary FSM transitions Asleep → Provisioning → Ready, the
# markReady() function publishes a PrimaryEvent{Type:"primary_ready", ...}
# on gw:primary:events. The emerg subscriber (Plan 06.6-06b cross-package
# adapter) consumes this and, if emerg.FSM.State() == EmergencyActive,
# fires force-destroy with shutdown_reason='pitfall_11_primary_took_over'.

T_PRIMARY_READY=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
echo "T_PRIMARY_READY=$T_PRIMARY_READY"

sleep 30

# Verify emerg has force-destroyed
EMERG_STATE_AFTER=$(docker exec ai-gateway-dev_gateway /gatewayctl emerg state --format=json | jq -r '.state // "healthy"')
echo "EMERG_STATE_AFTER=$EMERG_STATE_AFTER"
[ "$EMERG_STATE_AFTER" != "emergency_active" ] \
  && echo "ASSERTION PASS: emerg FSM left EmergencyActive within 30s of primary_ready (state=$EMERG_STATE_AFTER)" \
  || echo "ASSERTION FAIL: emerg still EmergencyActive after 30s — Pitfall #11 subscriber did NOT fire"

# Confirm emerg lifecycle row closed with the right shutdown_reason
docker exec ai-gateway-dev_gateway /gatewayctl emerg lifecycles --since 1h --format=json \
  | jq '.[0] | {id, ended: .EndedAt, shutdown: .ShutdownReason}'
# Expected: ended_at populated + shutdown_reason ~ "pitfall_11_primary_took_over"
# (exact reason string depends on the emerg subscriber implementation — Plan 06.6-06b).

# Confirm Vast UI: emerg instance destroyed, primary instance still running
echo "Verify in Vast UI: https://cloud.vast.ai/instances/"
echo "  - Emerg instance ID $EMERG_INST should be in 'destroyed' state"
echo "  - Primary instance should be active (state=running)"

# ============================================================
# CLEANUP: tear down primary
# ============================================================
docker exec ai-gateway-dev_gateway /gatewayctl primary force-down --reason "uat_pitfall11_cleanup"

# Wait drain + destroy
sleep $((300 + 60))  # grace_seconds + buffer

# Confirm both FSMs idle
docker exec ai-gateway-dev_gateway /gatewayctl emerg state --format=json
docker exec ai-gateway-dev_gateway /gatewayctl primary state --format=json
```

**Expected:**

- emerg pod provisions to `EmergencyActive` (typically 5-10min cold-start).
- primary pod provisions to `ready` (25-40min cold-start, ON TOP of emerg
  still running — temporary 30-40min window of dual pod billing is expected

**Pass/Fail criteria:**

  lifecycle row closed with pitfall_11 reason; Vast UI confirms only
  primary instance active during the overlap window.
- FAIL: emerg still `EmergencyActive` >30s after primary `ready`, OR emerg
  lifecycle row not closed, OR > 30s window of dual pod billing observed.

**Cleanup:** see CLEANUP block above. Verify ZERO active instances in Vast
