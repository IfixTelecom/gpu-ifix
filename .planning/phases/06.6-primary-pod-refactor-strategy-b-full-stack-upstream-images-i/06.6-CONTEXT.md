## Phase Boundary

Migrar o pod primário Ifix (Phase 1 — atualmente `ghcr.io/ifixtelecom/ifix-ai-pod` custom GHCR image, 5 serviços baked: llama.cpp + Speaches + Infinity + health-bridge Go + dcgm-exporter) para o mesmo pattern do emergency pod Phase 6 Strategy B: imagem base upstream + bootstrap bash inline (sem GHCR Ifix custom) + tag-pinned + MinIO weight fetch com sha256 verify + Runtype=args + Onstart=/bin/bash + Args=["-c", `<script>`].

**Trigger:** Phase 6 PR2 (commit 064702e) deletou pod/Dockerfile + .github/workflows/build-pod.yml + pod/scripts/emerg-bootstrap.sh achando que afetavam só emergency pod, mas o MESMO image custom era usado tanto por emergency QUANTO por primary pod (`pod/onstart.sh` + HOST-mode docker-compose Phase 1). Imagem GHCR cached `ghcr.io/ifixtelecom/ifix-ai-pod:develop-eaa6188` continua disponível, mas após operator GHCR delete (~2 semanas observability window) primary pod path quebra completamente. Phase 6.6 fecha esta gap arquitetural.

**Antes:**
- Image: `ghcr.io/ifixtelecom/ifix-ai-pod:{tag}` (~2GB binários baked + Phase 1 build-pod.yml CI)
- Bootstrap: `pod/onstart.sh` (executa dentro do image via Vast SSH injection) + `docker compose up` 5 services internamente
- Runtype: `ssh` (mesma classe de bug que Phase 6.5 sofreu — CMD-ignore caso onstart trave)

**Depois:**
- Image base: nvidia/cuda CUDA runtime upstream (tag-pinned) + supervisord
- Bootstrap: inline bash (raw-string Go const em `gateway/internal/primary/` ou similar) executado via Runtype=args + Onstart=/bin/bash + Args=["-c", `<script>`]
- Script faz: apt install supervisord + curl-download mc + download weights MinIO (Qwen + Whisper + BGE) com sha256 + pull/start upstream service images via supervisord-managed processes (OR docker-in-docker compose, decisão final via research)
- 4 services managed (LLM via llama.cpp + STT via Speaches + embed via Infinity + DCGM). Health-bridge **DROPPED** (gateway Phase 3 ProbeIntervalSeconds=10 substitui — duplicação eliminada).

**Não é desta phase:**
- Mudanças no gateway code (FSM, reconciler, fallback chain) — primary pod refactor é orthogonal
- Migrar para vLLM ou outro inference engine — llama.cpp server-cuda b9128 (Phase 6 pattern) é a base
- Multimodal Qwen3.6 vision endpoint — text-only inicialmente, vision endpoint = phase futura se ConverseAI/etc precisar
- Tear down Phase 1 docs/RUNBOOK secundários — pod/README.md atualizado para refletir nova arch

## Implementation Decisions

### Scope — Serviços no pod primário

- **D-01:** **5 serviços originais Phase 1 mantidos** (user override sobre Option simpler 2-service): LLM + STT + embed + DCGM exporter + ~~health-bridge~~ (HB dropped — ver D-05). Final shape: 4 serviços rodando + 1 dropped.
- **D-02:** Cada serviço usa **upstream image tag-pinned** (zero custom GHCR Ifix). Phase 6 D-01-B principle estendido ao stack completo.

### Engine + modelo LLM

- **D-03:** Engine LLM = **`ghcr.io/ggml-org/llama.cpp:server-cuda-b9128`** (mesma image que emergency pod Phase 6 — pattern já validado live UAT lifecycle 39). NÃO vLLM (template Vast.ai vLLM apenas inspiração visual; multimodal vision endpoint não é Phase 6.6 scope).
- **D-04:** Modelo LLM = **Qwen3.6-27B Q4_K_M GGUF** (`unsloth/Qwen3.6-27B-MTP-GGUF` upstream, file `Qwen3.6-27B-Q4_K_M.gguf`). Already uploaded em MinIO:
  - Key: `qwen3.6-27b-Q4_K_M/v1.0.0/model.gguf` (17.106 GB)
  - Sha256: `a7cbd3ecc0e3f9b333edee61ae66bc87ed713c5d49587a8355814722ed329e0f` (sidecar: `qwen3.6-27b-Q4_K_M/v1.0.0/model.gguf.sha256`)
  - Modo: **text-only** (multimodal vision endpoint = phase futura; mmproj sidecar não carregado em Phase 6.6).
- **D-04-alt:** Jinja tool-calling template para Qwen3.6 = decisão research-driven (verificar se Phase 6 `qwen3.5-27b-tool-calling.jinja` é compat OR se Qwen3.6 exige novo Jinja).

### Health probing — Drop health-bridge

- **D-05:** **Drop pod/health-bridge custom Go service**. Phase 3 gateway probe (`ProbeIntervalSeconds=10` em `gateway/internal/config/config.go`) já roda active healthchecks em cada upstream com latency tracking + breaker FSM. Duplicação eliminada.
  - Consequência: 0 custom GHCR images remanescentes pós-Phase 6.6.
  - Phase 5 saturation FSM ainda recebe inflight + P95 signal via gateway probe. VRAM signal (DCGM) preservado via D-07.

### Orchestration approach (5→4 services no pod)

- **D-06:** Tendência inicial = **supervisord hybrid** (Option 3) — base image nvidia/cuda + onstart apt-install supervisord + 4 procs (llama-server + Speaches + Infinity + dcgm-exporter) managed via supervisord.conf. PORÉM final decision **deferida ao researcher** após avaliar:
  - Viabilidade DinD em Vast.ai args runtype (privileged container support?)
  - Disponibilidade Speaches + Infinity como standalone binaries OR Python venv requirements
  - Trade-off supervisord (apt-installed) vs docker-in-docker (privileged) vs native bash background-procs
  - Vast.ai template `vLLM` (readme_hash `597d9d8832808e0e17ac64cf5eac641b`, template id 422675, image `vllm/vllm-openai:latest`, runtype=ssh, ports `1111:1111 8080:8080 8000:8000 8265:8265 8089:8089`, ENV `OPEN_BUTTON_PORT`/`PORTAL_CONFIG`/`USE_ALL_GPUS=true`) — não adotado literal, mas pattern PORTAL_CONFIG (Vast Instance Portal UI) pode ser reutilizado para ops debug se útil.

### DCGM exporter

- **D-07:** **Manter DCGM exporter** (port 9400) — upstream `nvcr.io/nvidia/k8s/dcgm-exporter:<tag-pinned>`. Preserva 3-signal Phase 5 saturation FSM (inflight + P95 + VRAM via DCGM scrape). Sem custom Dockerfile — só pull upstream image + run.

### Lifecycle trigger — Schedule-based via env vars

- **D-08:** **Schedule-based via env vars** (não 24/7, não pure on-demand). Pod up em peak hours, down em off-hours. Custo estimado ~$130/mo (~14h × 22 dias × $0.40).
- **D-08.1:** Env vars (nomes finais research-driven, sugestão inicial):
  - `PRIMARY_POD_SCHEDULE_TIMEZONE=America/Sao_Paulo` (default)
  - `PRIMARY_POD_SCHEDULE_UP_HOUR` (e.g., 8)
  - `PRIMARY_POD_SCHEDULE_DOWN_HOUR` (e.g., 22)
  - `PRIMARY_POD_SCHEDULE_DAYS` (CSV `mon,tue,wed,thu,fri`)
  - `PRIMARY_POD_SCHEDULE_GRACE_RAMP_DOWN_SECONDS` (drain inflight antes de destroy)
  - `PRIMARY_POD_SCHEDULE_DISABLED` (operator kill-switch)
- **D-08.2:** Off-hours behavior = apps caem em Phase 3 fallback chain (OpenRouter LLM + OpenAI Whisper STT + OpenAI embed). Já validado functionally — phases 3/5 fallback chain code não muda.
- **D-08.3:** Reconciler — extend emerg reconciler (Phase 6.5 architecture) OR novo primary reconciler. **Research decide** baseado em code reuse vs separation of concerns.

### Weight cache strategy

- **D-09:** Weights baixados toda lifecycle (mesmo pattern emergency Phase 6). **Vast persistent volumes + host pin = deferred** (arch follow-up; mesmo gap SC-2 que Phase 6 enfrentou — cold-start 20min dominado por 16-17GB WAN download).
- **D-09.1:** Workaround prático = pod schedule garante up só peak hours. Cold-start ~20min daily acceptable se primeira request peak hour vem >20min após scheduled-up time. Operator orquestra (pre-warm cron OR primeiro request triggers).

### Operator debug SSH

- **D-10:** **Reusar `POD_DEBUG_SSH_PUBLIC_KEY`** env var (Phase 6 commit 4896004 + 19a66a3). Mesma onstart inline sshd bootstrap pattern. Vast docker port mapping `-p 22:22` quando key set. Production: env vazio (least-privilege).

### Claude's Discretion

- supervisord vs DinD vs native multi-process bash final decision — researcher recomenda + planner implementa
- Schedule reconciler integration (new file vs extend emerg) — pattern-driven
- Speaches + Infinity exact upstream tags + env config — researcher pesquisa estabilidade tags
- Jinja Qwen3.6 template (if different from Qwen 3.5) — researcher confere modelo card unsloth + ggml-org guides

## Existing Code Insights

### Reusable Assets

- `gateway/internal/emerg/vast/types.go` — `CreateRequest` DTO já tem fields Image, Onstart, Runtype, Args, Disk, Env, Entrypoint (omitempty ignored by Vast). Primary pod reusa mesmo DTO.
- `gateway/internal/emerg/vast/client.go` — `CreateInstance`, `GetInstance`, `DestroyInstance`, `SearchOffers` HTTP client com retry/sentinel error mapping. Primary pod reusa cliente.
- `gateway/internal/emerg/lifecycle.go` patterns — `buildCreateRequest` shape (raw-string Go const onstart + env map + Vast offer→payload assembly); `provisionLifecycle` retry/destroy logic; FSM cooldown handling. Primary pod faz fork OU shared abstraction.
- `gateway/internal/config/config.go` envOr/csvOr helpers — primary pod fields seguem mesma convention.
- `gateway/internal/emerg/lifecycle_test.go` — 9 unit tests (TestBuildCreateRequest_StrategyB_args + variants). Primary pod plan deve replicar test pattern.
- `pod/scripts/upload-weights.sh` (preservado) — primary pod ainda usa mesma MinIO upload pipeline para atualizar weights pinned.

### Established Patterns

- **Strategy B payload shape** (Phase 6 D-06-B + D-07-B revised): `Runtype="args"` + `Onstart="/bin/bash"` + `Args=["-c", emergencyOnstart]` + Entrypoint omitido (Vast API has no entrypoint field). Primary pod payload é struturalmente idêntico, só script content + env vars diferem.
- **Inline onstart raw-string Go const** (`emergencyOnstartHead` em lifecycle.go:694) — Pitfall 9 RESEARCH.md:476 enforced (zero fmt.Sprintf). Primary pod onstart segue mesmo pattern, mas mais longo (~3-5KB vs 1015 chars emergency).
- **MinIO weight fetch + sha256 verify** via mc CLI (curl-installed inline, sem apt). Phase 6 commit 4896004 estabeleceu pattern. Primary pod consome múltiplos files (qwen+whisper+bge) com mesma técnica.
- **Phase 3 gateway probe loop** (ProbeIntervalSeconds=10, per-upstream breaker FSM) — substitui health-bridge pod-side. Gateway boot já wire isto. Primary pod deploy = gateway começa a observar Vast pod URL como healthy local-llm/local-stt/local-embed upstreams.

### Integration Points

- **Reconciler:** primary pod precisa controle lifecycle (provisionar peak hour, destroy off-hour). Decisão D-08.3: extend `gateway/internal/emerg/reconciler.go` OR criar `gateway/internal/primary/reconciler.go`. Researcher decide baseado em separation of concerns.
- **Schedule clock:** novo subsystem dentro reconciler (time.Now() vs cfg schedule comparison) + go-cron-like trigger.
- **Upstream URL discovery:** quando primary pod sobe, gateway precisa atualizar `local-llm.URL` para `http://<vast_host>:<mapped_8000>` em runtime. Existem patterns Phase 2 (`UpstreamsListenAndReload` LISTEN/NOTIFY) — primary reconciler emite NOTIFY após pod ready, gateway recarrega.
- **Drain ramp-down:** ao schedule down hour, reconciler marca primary upstream "draining" → gateway redireciona requests novas para fallback chain → após `PRIMARY_POD_SCHEDULE_GRACE_RAMP_DOWN_SECONDS` destroy pod.
