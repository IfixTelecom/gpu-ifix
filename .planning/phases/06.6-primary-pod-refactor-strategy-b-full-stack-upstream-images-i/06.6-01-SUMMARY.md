## Tasks executed

### Task 1 — DinD privileged + GPU pass-through spike

**Result:** **FAIL** (final spike verdict at `06.6-SPIKE-dind-privileged.md`).

4 instances across 4 hosts (DE/NO/IS) tested progressive workarounds:

1. Raw DinD on DE → host docker rejected `nvidia` runtime + `--privileged` combo.
2. Raw DinD on NO → dockerd booted, but `iptables nf_tables: Permission denied` blocked bridge init.
3. DinD + `iptables=false + bridge=none` on IS → host "Internal error" (vague).
4. Same workaround on NO → dockerd booted cleanly, BUT **overlayfs mount in nested namespace failed `operation not permitted`** at image extraction. Fundamental Linux kernel namespace limitation, not bypassable.

**Re-eval discovery:** Strategy B's intent (4 services co-located in 1 pod) is **still viable via supervisord** as PID 1 with 4 child processes. DinD-specific approach is dead; supervisord-based co-location replaces it.

### Task 2 — Qwen3.6 embedded Jinja tool-calling spike

**Result:** **PASS** (final verdict at `06.6-SPIKE-qwen3.6-jinja.md`).

Iterative path:
- Round 1: `mc cp` single-stream Qwen3.6 GGUF download FAILED twice with `unexpected EOF` on UK pod (long-lived TCP fragility).
- Round 2: Rewrote onstart to `aria2c -x 16 --continue` multi-stream — download PASS (sha256 MATCH `a7cbd3ec...`). BUT llama-server with `b9128` image FAILED at model load: `missing tensor 'blk.64.ssm_conv1d.weight'`. Qwen3.6 uses hybrid Transformer+SSM architecture; b9128 too old.
- Round 3: Upgraded image tag to `server-cuda-b9191` (includes recent SSM/recurrent model PRs #23121, #22384). Same Qwen3.6 GGUF + same `--jinja` flag → llama-server boots, `chat_format: peg-native`, tool-call test returns valid `tool_calls=[{name:"get_weather", arguments:{location:"São Paulo"}}]`. Generation 770 tok/s prompt + 47 tok/s predict on RTX 4090.
