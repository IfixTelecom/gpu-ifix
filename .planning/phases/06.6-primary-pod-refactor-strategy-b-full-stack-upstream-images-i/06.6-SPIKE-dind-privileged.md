## Round 2 — Raw DinD, offer 36539953 (NO, $0.54/h, driver 565)

**Result:** Container started successfully, dockerd entered initialization, but failed at **bridge network controller init**:

```
=== DOCKERD-LOG-TAIL ===
time="..." level=info msg="Starting up"
time="..." level=warning msg="Error while setting daemon root propagation, ...
  could not setup daemon root propagation to shared:
  mount /var/lib/docker:/var/lib/docker, flags: 0x1000: operation not permitted"
... (containerd boots cleanly) ...
time="..." failed to start daemon: Error initializing network controller:
  error obtaining controller instance: failed to register "bridge" driver:
  failed to create NAT chain DOCKER: iptables failed:
  iptables --wait -t nat -N DOCKER:
  iptables v1.8.10 (nf_tables): Could not fetch rule set generation id:

  (exit status 4)
```

**Finding:** Even with `--privileged:1` accepted at Vast envelope level, **nf_tables kernel operations are blocked in the container namespace**. The container has CAP_NET_ADMIN nominally, but lacks the namespace-level capability to manipulate the host's iptables/nftables rule set generation IDs.

Instance 36940950 destroyed. Cost ~$0.054.

## Round 3 — DinD with `iptables=false + bridge=none` workaround, offer 9021757 (IS, $0.40/h)

**Onstart v2:** Wrote `/etc/docker/daemon.json` with `{"iptables": false, "bridge": "none", "ip-forward": false}` BEFORE `dockerd` start. Nested containers planned `--network=host`.

**Result:** Container never reached `running`. Vast.ai reported `status_msg: "Error: Internal error."` (vague host-side rejection). Instance 36942297 destroyed at `actual_status=loading`. Cost ~$0.030.

**Result:** Container started, dockerd booted SUCCESSFULLY with iptables disabled, BUT **image extraction failed at overlayfs mount layer**.

dockerd boot log (success):

**Root cause:** **overlayfs mount/unmount in nested namespace fails with `operation not permitted`** even with `--privileged`. This is a fundamental Linux kernel namespace limitation. Containers' overlayfs snapshotter cannot perform the mount operations needed to extract layered images.

Instance 36942570 destroyed. Cost ~$0.090.

## Spike Result — DinD FAIL

### **FAIL — DinD with `--privileged:1` NOT VIABLE on Vast.ai**

**Cumulative findings:**

1. `--privileged:1` IS accepted by Vast.ai env field (3 of 4 hosts).
2. With privileged, dockerd CAN boot inside container BUT iptables/nf_tables manipulation is blocked (Round 2).
3. Workaround `iptables=false + bridge=none` resolves dockerd network init (Round 4), BUT:
4. **overlayfs mount/unmount in nested namespace fails with `operation not permitted`** — fundamental kernel namespace limitation, not bypassable with flag config.

Alternative storage drivers (`vfs`, `fuse-overlayfs`) work in unprivileged namespaces but: VFS is ~10x slower image extraction; fuse-overlayfs adds dependency + perf hit. For a primary pod launching multiple ~1.5GB+ upstream images, VFS-based DinD makes cold-start prohibitively slow.

## Strategy Re-evaluation — Vast.ai is Container Hosting, Not VM Hosting

**Important clarification (post-spike):** Vast.ai sells container slots, NOT VPS/VM rentals. You rent a Docker container running on a shared host. Therefore:

- `--privileged` is host-policy-restricted (Vast can't grant true host root).
- DinD/nested Docker doesn't work on most hosts.
- **BUT** multi-process inside a single container DOES work via standard process supervisors (supervisord, runit, s6, systemd-style init).

This means **Strategy B's intent (co-locate 4 services in one pod)** is still viable — via **supervisord** as PID 1 with 4 child processes, NOT via DinD with 4 nested docker containers.

### Strategy B-revised — supervisord co-location

```
PID 1 = supervisord
  ├─ llama-server  (Qwen3.6 LLM, port 8000)
  ├─ speaches      (Whisper STT, port 8001)
  ├─ infinity      (BGE-M3 embeddings, port 8002)
  └─ dcgm-exporter (GPU metrics, port 9400)
```

All 4 children share:
- Single container's network namespace → talk via `localhost:<port>` between siblings.
- Vast.ai `-p 8000:8000 -p 8001:8001 -p 8002:8002 -p 9400:9400` exposes each externally.
- GPU device — RTX 4090 visible to all via standard `/dev/nvidia0` + CUDA. (Whisper/BGE-M3 can both use CUDA simultaneously; llama.cpp uses VRAM for Qwen3.6 27B Q4_K_M ~17GB; speaches Whisper-Medium ~3GB VRAM; BGE-M3 ~2GB VRAM → fits comfortably in 24GB RTX 4090.)
- Filesystem — shared `/weights/`, `/models/`, etc.

**Implementation strategy options:**

- **α (custom image):** Build a multi-stage image bundling all 4 binaries + supervisord. Violates D-08-B "no custom image" rule from CONTEXT.md. Engineering-cleanest path.
- **β (runtime install):** Use `ghcr.io/ggml-org/llama.cpp:server-cuda-b9191` as base. Onstart `apt install supervisor` + download speaches/infinity binaries (apt or pip or pre-built tarballs from MinIO) + write supervisord config. Fragile but respects D-08-B.
- **γ (hybrid):** Use upstream Speaches Docker image as base (it has Python + uvicorn already), supervisord-install at runtime, then drop in llama-server + infinity binaries. Probably worst — debian deps clash.

Plan 06.6-04 (`primary/onstart.go`) and CONTEXT.md D-08-B should pick α or β. The empirical data from spikes does NOT prefer one over the other — both are viable.
