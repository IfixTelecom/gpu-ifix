# Cross-AI Plan Review — Phase 06.6

## Gemini Review

# Phase 06.6 Plan Review: Primary Pod Refactor (Strategy B)

This review evaluates the 12-plan sequence for refactoring the primary inference pod to the **Strategy B Full-Stack** pattern.

## 1. Summary
The plan set is **exceptional in detail and architectural consistency**. It successfully adapts the proven Strategy B pattern from the emergency pod to the more complex, multi-service primary pod. By extracting shared utilities into `vastutil` and maintaining strict separation between emergency and primary logic, the plans avoid technical debt while significantly hardening the system. The "Wave 0" approach to spike high-risk assumptions (DinD and Qwen3.6 templates) is a textbook example of proactive risk management.

## 2. Strengths
*   **Surgical Refactoring:** Plan 06.6-02 extracts `vastutil` helpers from the `emerg` package. This prevents code duplication and establishes a clean "infrastructure utility" layer for future pods.
*   **State Machine Robustness:** The introduction of a 5-state FSM (Plan 06.6-05) specifically designed for schedule-driven lifecycles—including a dedicated `Draining` state—is superior to a naive "up/down" toggle. This directly supports the **Core Value** of invisible failover by allowing in-flight requests to complete during off-hour transitions.
*   **Inter-reconciler Coordination:** Plan 06.6-08 specifically addresses **Pitfall #11** (emerg/primary race). The "Primary Precedence" logic (Option B) is correctly implemented via Redis Pub/Sub, ensuring that the system converges on a single pod without manual intervention.
*   **Zero-Custom Image Goal:** The plans successfully eliminate the last dependency on the custom `ifix-ai-pod` GHCR image, significantly reducing the maintenance surface and build pipeline complexity.
*   **Security & Integrity:** Mandatory SHA-256 manifest pinning (Plan 06.6-01) and weights verification during bootstrap maintain high supply-chain security standards.

## 3. Concerns

### [HIGH] DinD (Docker-in-Docker) Dependency
*   **Risk:** The 4-service primary stack relies on `dockerd` running inside a privileged Vast.ai container (D-06). Vast.ai host support for `--privileged` is unverified.
*   **Impact:** If the Wave 0 spike (Plan 01, Task 1) fails, the entire multi-service strategy is blocked, forcing a retreat to "Strategy E" (LLM-only primary), which would regress the STT and Embedding capabilities to external-only during peak hours.

### [MEDIUM] Context Budget for Plan 06
*   **Risk:** Plan 06.6-06 is a "mega-plan" modifying 8 files across different packages (`dcgm`, `shed`, `upstreams`, `primary`).
*   **Impact:** While cohesive, the sheer volume of changes may exceed the context window for a single execution session. The "Scope Note" in Plan 06 correctly advises task-level checkpoints, but the executor must be disciplined in committing between tasks to avoid runaway context spend.

### [MEDIUM] Cold-Start "Known Gap"
*   **Risk:** The 25-30 minute cold-start (Pitfall #3) is documented as an accepted gap, but it creates a "dead zone" at the start of every peak window.
*   **Impact:** If peak is set to 08:00, the pod won't serve until ~08:30. While the plan suggests "operator orchestration" (setting UP_HOUR to 07:00), the lack of an automated "pre-warm" offset in the code makes the system reliant on manual schedule adjustments.

### [LOW] Migration Numbering Drift
*   **Risk:** Plan 06.6-07 uses migration `0023`.
*   **Impact:** If any concurrent "quick tasks" or other branches land migrations in the meantime, this will cause a conflict. The plan includes an update to `migrate_test.go`, which is good, but the executor must verify the latest sequence number at the moment of Task 1 execution.

## 4. Suggestions
*   **Schedule Pre-warm Logic:** Consider adding a `PRIMARY_POD_SCHEDULE_PREWARM_MINUTES` env var (default 30) that triggers provisioning *before* the `UP_HOUR`, so the pod is `Ready` exactly when peak begins.
*   **Vast status_msg Check:** Incorporate the fix for `status_msg` detection mentioned in the `STATE.md` open todos (lifecycle 29 forensics) into Plan 06.6-06. Since the reconciler is being built fresh, it is the perfect time to add `if inst.StatusMsg contains "Error" -> Abort`.
*   **GHA Smoke Workflow:** In Plan 06.6-12, instead of simply deleting `.github/workflows/smoke.yml`, consider a follow-up task to create a new "Gateway-driven Smoke" that triggers the `primary.Reconciler` via `gatewayctl` rather than manual `vast-ai.sh` calls. This preserves the "CI-verified" nature of images.

## 5. Risk Assessment: MEDIUM
The overall risk is **MEDIUM**, driven almost entirely by the **DinD/Privileged Container** assumption. From a software engineering perspective, the plans are **Low Risk** due to the high reuse of patterns from the already-validated Emergency Pod refactor. If the Plan 01 spikes pass, the rest of the phase is a highly deterministic "construction" phase with excellent test coverage.

**Approval Recommendation:** Approved to proceed to Wave 0. The results of 06.6-01-PLAN are critical path milestones.

## Codex Review

## Summary

The 12-plan set is unusually thorough and generally well-sequenced: it identifies the right high-risk areas up front, gates destructive cleanup behind live UAT, and keeps the core phase goal visible: remove the custom `ifix-ai-pod` dependency, replace it with Strategy B upstream images plus inline bootstrap, and add schedule-driven primary lifecycle without breaking the emergency reconciler. The biggest risks are not conceptual but executional: Plan 06 is too large, several tests rely on internals or planned APIs that may not exist cleanly, the DinD production posture is accepted rather than hardened, and the schedule/drain/cutover behavior may be more complex than necessary for a primary pod refactor phase.

## Strengths

- **Good burnt-bridge sequencing.** Plans 01 and 11 gate unknowns and live UAT before Plan 12 deletes legacy `pod/` artifacts and `.github/workflows/smoke.yml`. This is the right shape for replacing a production bootstrap path.

- **Wave 0 addresses real unknowns.** `06.6-01` correctly validates DinD/privileged support, GPU pass-through, Qwen3.6 tool-calling template behavior, and image digest pinning before implementation.

- **Supply-chain posture is improved.** SHA pin capture for the 4 upstream images in `06.6-01`, combined with later config defaults, is materially better than mutable tags or custom GHCR image drift.

- **Emergency coexistence is explicitly handled.** Pitfall #11 appears in `06.6-08`, `06.6-10`, and `06.6-11`, with primary precedence, Pub/Sub handling, integration tests, and live UAT.

- **Health-bridge removal is coherent.** Dropping the custom health-bridge per D-05 and relying on gateway-side probes is a clean simplification, provided the probe matrix is actually complete for LLM/STT/embed/DCGM.

- **Operator UX is considered.** `gatewayctl primary`, Redis state, lifecycle table, runbook, and UAT docs give operators a workable control surface.

- **PR3 cleanup is correctly delayed.** `06.6-12` only runs after live UAT sign-off and preserves useful artifacts such as `upload-weights.sh`, `vast-ai.sh`, templates, and smoke fixtures.

## Concerns

- **HIGH — Plan 06 is too large and too coupled.**
  `06.6-06` modifies the reconciler, budget logic, upstream loader, DCGM scraper, inflight registry, and tests in one plan. The plan itself acknowledges the checker warning but keeps the scope. This creates a high merge/debug risk because failures can arise from several unrelated mechanisms: dynamic tier-0 routing, drain polling, DB lifecycle state, redsync, budget aggregation, and DCGM URL mutation.

- **HIGH — DinD privileged production posture is accepted too casually.**
  `06.6-01`, `06.6-04`, and `06.6-11` treat privileged DinD as acceptable after a spike. That may be operationally pragmatic, but the plans do not sufficiently constrain the blast radius: MinIO credentials, SSH debug, Docker socket, image pulls, and GPU access all coexist in a privileged container. The security model is mostly “short-lived” for the spike and “Vast tenant isolation” for production.

- **HIGH — Schedule behavior may be wrong for overnight windows and day filters.**
  In `06.6-05`, `IsInPeak` checks `r.Days[now.In(tz).Weekday()]` before applying overnight logic. For a schedule like `mon 22 -> tue 08`, Tuesday 02:00 should arguably be part of Monday’s active window if Monday is enabled. The proposed implementation checks Tuesday’s day instead, which changes semantics. This matters if default ever changes from 08-22 to overnight.

- **HIGH — Plan 08 assumes `PRIMARY_POD_SCHEDULE_DISABLED=true` makes `Start` return immediately, but Plan 06 also needs force-up.**
  `06.6-08` says Start returns immediately if disabled. `06.6-09` adds `force-up`, and `06.6-11` starts with disabled=true then runs `gatewayctl primary force-up`. If Start exits when disabled, force-up events may never be consumed unless a separate command subscriber remains running. This is a real semantic conflict.

- **HIGH — `force-up` / `force-down` event handling is under-specified.**
  `06.6-09` publishes `force_up_request` and `force_down_request`, but `06.6-06` does not clearly implement a primary event subscriber that handles those events. `06.6-08` wires only `primary.Reconciler.Start`, while the event subscriber described is for the emergency reconciler consuming `primary_ready`. The CLI may publish events that nobody processes.

- **MEDIUM — Tests depend heavily on unexported internals and speculative APIs.**
  `06.6-10` calls `markReady`, checks `isLeader`, mutates `r.rule`, and expects test-only helpers. This may be acceptable in Go package tests, but the integration tests are in `gateway/internal/integration_test` package, not `primary`, so direct access to unexported fields/methods will not compile unless exported wrappers are explicitly added. `export_test.go` inside `primary` is not available to a different package’s tests unless those tests import test-only symbols, which Go does not allow across packages in that way.

- **MEDIUM — sqlc generation is hand-waved.**
  `06.6-07` says regenerate sqlc if needed, and if unavailable document it. But `06.6-06` depends on generated bindings. This should be a hard prerequisite with a verified command, not optional documentation.

- **MEDIUM — Migration numbering/count language is inconsistent.**
  `06.6-07` creates `0023_primary_lifecycles.sql` but says update migration count `19 -> 20` and includes a confusing comment about “0001..0023 = 23 files but 02 split”. This is likely to cause test churn. The plan should inspect actual migrations and set the expected count exactly.

- **MEDIUM — `onstart.go` uses shell env expansion without adequate quoting.**
  `06.6-04` says dynamic values flow via `$VAR`, but the plan does not require robust shell quoting around MinIO keys, endpoint, bucket, or image values. Operator-controlled values are trusted, but malformed env values can still break bootstrap or cause surprising shell behavior.

- **MEDIUM — MinIO SHA policy is incomplete for Whisper/BGE.**
  `06.6-03` explicitly leaves `PRIMARY_WHISPER_WEIGHTS_SHA256` and `PRIMARY_BGEM3_WEIGHTS_SHA256` empty by default, while `06.6-04` requires sha256 verify for all 3 downloads. If unset, the script behavior needs to be defined: fail fast, skip verification with loud warning, or refuse to create request. Current plans conflict between “verify every download” and “no default shipped”.

- **MEDIUM — Cold-start remains a major product risk.**
  The plans acknowledge 20-30 minute cold starts but mostly accept them. For a schedule-based primary pod, that means `UP_HOUR=8` may not be usable until 08:25-08:40. The UAT verifies behavior but does not ensure operator-facing schedule semantics include pre-warm offset.

- **MEDIUM — Runtime lifecycle state is split across DB, Redis, FSM, loader overrides, and DCGM URL.**
  The design is workable, but recovery after gateway restart is not sufficiently covered. If the gateway restarts while a primary pod is already running, plans do not clearly describe how it rehydrates `activeInstanceID`, `activePodURLs`, lifecycle ID, loader overrides, and DCGM URL.

- **MEDIUM — Main gateway probe integration is vague.**
  The phase goal says health-bridge is replaced by Phase 3 probes. But `06.6-06`/`06.6-08` should state exactly how local upstream definitions become the dynamic pod URLs and how probe targets update for all 3 roles. “LISTEN/NOTIFY or direct Loader.Refresh” is too loose for a critical path.

- **LOW — Plan 04 lifecycle payload tests use defaults but config defaults may be SHA-pinned later.**
  Some tests assert `req.Image == "ghcr.io/ggml-org/llama.cpp:server-cuda-b9128"` while Wave 0 requires `@sha256` pins. Decide whether defaults are tag-only or digest-pinned. The current wording says both.

- **LOW — `dcgm.Scraper.SetURL` implementation choice may be invasive.**
  Replacing `url string` with `atomic.Pointer[string]` can ripple through existing tests and methods. A mutex may be simpler and less error-prone unless scrape is extremely hot.

- **LOW — Deleting `.github/workflows/smoke.yml` may remove useful automated smoke coverage.**
  `06.6-12` argues the workflow is legacy and tied to deleted artifacts. That is plausible, but a replacement scheduled/manual smoke workflow for Strategy B would be valuable later.

## Suggestions

- Split `06.6-06` into two plans:
  - `06.6-06a`: primary reconciler state machine, lifecycle, drain, cooldown.
  - `06.6-06b`: external adapter changes: loader 3-role overrides, DCGM `SetURL`, inflight `Count`, budget.
  This reduces blast radius without changing the architecture.

- Resolve disabled/force-up semantics before implementation. Recommended behavior:
  - `PRIMARY_POD_SCHEDULE_DISABLED=true` disables automatic schedule ticks only.
  - The reconciler still starts and listens for `force_up_request` / `force_down_request`.
  - Add explicit tests: disabled schedule plus force-up provisions; disabled schedule without force-up stays asleep.

- Add a primary command/event subscriber plan item. `gatewayctl primary force-up` and `force-down` need a clear consumer in `primary.Reconciler`, not only Redis publishing.

- Define restart recovery behavior. Add a plan or tasks covering:
  - Find open `primary_lifecycles` row at boot.
  - Query Vast instance if `vast_instance_id` exists.
  - Rebuild pod URLs from mapped ports.
  - Restore loader overrides and DCGM URL if pod is healthy.
  - Close stale lifecycle if pod no longer exists.

- Tighten schedule semantics. For overnight schedules, decide whether early-morning hours belong to the previous scheduled day or the current day. Add tests for `mon 22 -> tue 08` with only Monday enabled.

- Make sqlc generation mandatory in `06.6-07`. Include the exact command, expected generated files, and a failing verification if bindings are missing.

- Make MinIO SHA handling explicit. Prefer fail-fast if Whisper/BGE SHA env vars are empty, since the plan’s stated security model depends on verifying all weights.

- Add shell hardening requirements to `primaryOnstartHead`:
  - quote all env vars,
  - validate required env vars before downloads,
  - fail with explicit messages,
  - avoid printing secrets under `set -x`, or temporarily disable xtrace around credential use.

- Reconsider `set -euxo pipefail` with secrets. `set -x` can leak MinIO access keys in pod logs when running `mc alias set`. Use `set -euo pipefail`, or wrap secret commands with `set +x` / `set -x`.

- Add a pre-warm offset config:
  - `PRIMARY_POD_SCHEDULE_READY_BY_HOUR=8`
  - `PRIMARY_POD_SCHEDULE_PROVISION_LEAD_SECONDS=2400`
  This reflects actual cold-start behavior better than “up at 8 means usable at 8”.

- Replace or later recreate `.github/workflows/smoke.yml` as Strategy B-compatible. Deleting the stale workflow is fine, but track a follow-up for automated non-GPU bootstrap validation or a manual workflow that calls `gatewayctl primary force-up` in a controlled environment.

## Risk Assessment

**Overall risk: HIGH.**

The architecture is coherent and the plan quality is strong, but the implementation surface is large and operationally sensitive: privileged DinD on Vast.ai, dynamic multi-role routing, schedule lifecycle, drain semantics, Redis leader election, DB lifecycle audit, emergency reconciler coexistence, and live GPU UAT all land in one phase. The most serious correctness risks are the disabled-vs-force-up conflict, missing primary event consumer for `force_up_request`, restart recovery gaps, and overnight schedule semantics. With those fixed and Plan 06 split, the risk drops to **MEDIUM**; without them, the plan could pass many unit tests while failing operator workflows or restart scenarios in production.
