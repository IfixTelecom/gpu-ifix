## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing Critical Functionality] `handleForceDownRequest` Provisioning branch**

- **Found during:** Task 2 (primary_cancel_inflight_test.go design)
- **Issue:** The plan describes a window-closes-mid-Provisioning auto-cancel scenario. But the production code's `evaluateTick` has a no-op `StateProvisioning` case (the schedule loop only observes the active lifecycle), AND `cancelActiveLifecycle` was DEFINED but UNREACHABLE — no event handler dispatched to it. The function's own doc comment says "Used by operator force-down while mid-provisioning", but `handleForceDownRequest` skipped that case.
- **Fix:** Wired `handleForceDownRequest` to dispatch to `cancelActiveLifecycle` when `r.deps.FSM.State() == StateProvisioning`. Operators can now publish `force_down_request` to abort a stuck provisioning lifecycle without waiting for the cold-start budget to expire. The Ready/Draining branches continue to call `startDrain` unchanged.
- **Files modified:** `gateway/internal/primary/reconciler.go`
- **Commit:** f675a3c

**2. [Rule 3 — Blocking Refactor] Extract `runPrimaryLifecyclesWithPool`**

- **Found during:** Task 3 (primary_lifecycles_integration_test.go design)
- **Issue:** The plan calls for integration coverage of the gatewayctl primary lifecycles DB-fetch path, but the existing `runPrimaryLifecycles` opens its own `*pgxpool.Pool` via `loadAndPool` which reads env vars. Threading a testcontainers DSN through env vars at test time is fragile (the env mutation is global; concurrent tests in the same package would race on the env state).
- **Fix:** Extracted `runPrimaryLifecyclesWithPool(ctx, pool, dur, limit, format) int` — the same body as before but accepts an already-built pool. Production `runPrimaryLifecycles` delegates to the new helper after flag parsing. External contract preserved.
- **Files modified:** `gateway/cmd/gatewayctl/primary.go`
- **Commit:** 42c713e

**3. [Rename — Symbol Collision] Integration test renamed**

- **Found during:** Task 3 build
- **Issue:** The unit test `primary_test.go` already declared `TestRunPrimaryLifecycles_FetchesFromDB` as a `t.Skip` placeholder. Under the `integration` build tag both files are part of the same test binary, producing a duplicate-symbol vet error.
- **Fix:** Renamed the integration test to `TestRunPrimaryLifecyclesIntegration_FetchesFromDB`. Documented the rename in the test's doc comment so the relationship between the t.Skip pointer and the integration test is explicit.
- **Files modified:** `gateway/cmd/gatewayctl/primary_lifecycles_integration_test.go`
- **Commit:** 42c713e

## Key Decisions

1. **Public-API-driven test design** — every integration test exercises the reconciler through `Start(ctx)` rather than calling private `evaluate*`/`markReady` methods. Assertions read from the test-owned FSM/Loader/DCGM/DB state. This is the only viable boundary because the integration_test package is a separate Go package and `export_test.go` symbols are not visible across packages.

2. **Wave 0 supervisord invariants via per-URL HealthCheck mocks** — each test wires a `func(ctx, url string) bool` closure that routes by host-port substring matching (`strings.Contains(url, ":33000")` etc.). This simulates the 4 supervisord children individually without standing up real supervisord — that's Live UAT scope (Plan 06.6-11).

3. **Pitfall #11 cross-package handoff coverage** — `primary_emerg_coexist_test.go` drives BOTH the emerg reconciler (via the existing emerg_provision_happy pattern) AND the primary publish path (direct `redisx.PublishPrimaryEvent`). The full handoff (gw:primary:events Pub/Sub → emerg.SubscribePrimaryEvents → emerg.cancelActiveLifecycle) is mechanically verified.

4. **`primary_overnight_schedule_test.go` as a pure-function test** — no testcontainers, no miniredis, no DB. The 4 wrap-day-filter scenarios are direct `ScheduleRule.IsInPeak(now)` invocations. The `integration` build tag keeps them grouped with the other primary tests, but the build cost is trivial.

5. **`handleForceDownRequest` Provisioning branch as the cancel-in-flight trigger** — the plan's `SetScheduleRuleForTest + trigger evaluateTick` approach is not viable from an external test package (evaluateTick is private + no Provisioning branch exists in production). The cleaner path is to drive cancel via `force_down_request` Pub/Sub from StateProvisioning, which is now reachable thanks to the Rule 2 fix.
