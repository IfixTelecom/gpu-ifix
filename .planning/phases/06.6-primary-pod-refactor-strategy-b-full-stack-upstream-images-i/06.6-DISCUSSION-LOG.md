### Area 2 — GPU offload check strictness

| Option | Description | Selected |
|--------|-------------|----------|
| Abort on CPU fallback | nvidia-smi pre-check + post-llama-boot CUDA backend verification + VRAM allocation check; kill llama + exit 1 if any fail | |
| Warn but continue | Same checks, but log warning + continue if CPU fallback. Gateway probe handles latency-level breaker. | |
| Skip explicit check, rely on gateway probe | Onstart unchanged. Gateway Phase 3 probe (ProbeIntervalSeconds=10, P95 breaker FSM) detects degraded latency naturally. | ✓ |

**Rationale:** Existing gateway probe mechanism already monitors latency P95 with breaker FSM. CPU-fallback degradation triggers high latency → breaker open → fallback chain redirect. No Plan 06.6-04 instrumentation overhead needed. Acceptable 5-10s detection delay.

### Area 3 — VRAM budget allocation

| Option | Description | Selected |
|--------|-------------|----------|
| Tight fit — full quality | Qwen3.6 17GB + Whisper-Large-v3 3GB + BGE-M3 full 2GB + KV 2GB = ~24GB exact. Zero margin. OOM risk under peak KV growth. Max quality. | ✓ |
| 2.5GB margin via Whisper-Medium + BGE-M3 INT8 | Downgrade STT/embed; OOM-safe; STT quality medium. | |
| KV cache cap menor (8k ctx) | Full quality models, half KV. Conversa longas truncam. | |

**Rationale:** Full quality preferred. Researcher proposes initial `--keep` / `--n-keep` flags to cap KV growth. Live UAT Plan 06.6-11 validates; if empirical OOM observed, env-override path to Whisper-Medium + BGE-M3 INT8 documented in deferred ideas.
