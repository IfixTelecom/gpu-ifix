## Summary

Phase 6.6 estende o pattern Strategy B (validado live UAT lifecycle 39 em Phase 6, 2026-05-17) do emergency pod LLM-only (1 serviço, 1015 chars onstart) para o **primary pod multi-serviço** (4 serviços, ~3.5-4.5KB onstart estimado). A refactor elimina o último Dockerfile custom GHCR (`ghcr.io/ifixtelecom/ifix-ai-pod`), substitui health-bridge custom por probe loop gateway-side já existente (Phase 3 `ProbeIntervalSeconds=10`), e adiciona um lifecycle controller schedule-based ortogonal ao reconciler emergency.

A maior diferença de risco vs Phase 6 é a **orquestração multi-processo**: Phase 6 só roda `exec /app/llama-server` como PID 1; Phase 6.6 precisa 4 processos cooperando dentro do container. A decisão D-06 (supervisord hybrid vs DinD vs native bash) é o crux research-driven. **Recomendação primária**: native bash background-procs com `wait -n` + trap (sem apt, sem privileged), porque (1) o llama.cpp:server-cuda base image NÃO traz supervisord e adicionar via apt aumenta cold-start em 15-30s + risco de debconf hang (Phase 6 hot-fix `4896004` já queimou esta caixa), (2) DinD precisa privileged + sidecar dockerd processo, custando ~200MB RAM + 5-10s extra, e (3) o número de serviços (4) é baixo o suficiente que um supervisor mínimo bash atende; supervisord só ganharia em ≥6 serviços ou crash-restart granular por processo. (Ver §Decisions Resolved item 1 para o trade-off completo + opção B/C como fallback.)

**Primary recommendation:** Implementar Phase 6.6 como **3 PRs sequenciais (PR1 config + lifecycle + scaffolding) → (PR2 schedule reconciler + LISTEN/NOTIFY wiring + UAT live) → (PR3 cleanup pod/Dockerfile + health-bridge + build-pod.yml + RUNBOOK)**. O pattern Strategy B já está provado; a complexidade nova é a orquestração 4-processos no onstart + o schedule reconciler. PR1 ships as code only (no live UAT), PR2 ships live UAT como gate para PR3 burnt-bridge cleanup. Mesma sequência burnt-bridge mitigation de Phase 6 (PR1 inline + PR2 delete pós-UAT GREEN).

### Pattern 1: Inline Bash Bootstrap (Strategy B)

**What:** Single bash string (raw-string Go const) é o conteúdo do `Args=["-c", <onstart>]` da `vast.CreateRequest`. Script faz weights fetch + multi-service start dentro do container. Padrão é uma adaptação do `emergencyOnstartHead` (lifecycle.go:694) para 4 serviços.

**When to use:** Sempre que Vast.ai pod precisa rodar serviços sem pre-baked image custom.

**Example (skeleton — full content em §Onstart Script Anatomy):**

```go
// Source: gateway/internal/emerg/lifecycle.go:694 (pattern reference)
// File: gateway/internal/primary/onstart.go (new)
const primaryOnstartHead = `set -euxo pipefail
export DEBIAN_FRONTEND=noninteractive
mkdir -p /weights/qwen /weights/whisper /weights/bge-m3 /app/templates
# ... (see §Onstart Script Anatomy for full ~3.5KB script)
`

func buildPrimaryOnstart(llamaArgs []string, ...) string {
    return primaryOnstartHead + "\n" + spawnServicesBlock + "\n" +
           "exec /app/llama-server " + strings.Join(llamaArgs, " ") + "\n"
}
```

### Pattern 2: 5-State Schedule FSM (Primary Lifecycle)

**What:** Reconciler com 5 states evaluados a 1Hz:
- `Asleep` — fora de schedule window, sem pod
- `Provisioning` — dentro de window, pod sendo criado (cold-start ~20min)
- `Ready` — pod healthy + tier-0 override ativo
- `Draining` — schedule down hour fired, redirecting new requests to fallback, waiting for inflight drain
- `Destroying` — drain complete, DestroyInstance issued

**When to use:** Primary pod precisa orquestração lifecycle distinta do emergency (que é OPEN-breaker-triggered, não schedule-triggered).

**Example transitions:**

```
Asleep ──── peakWindow.contains(now) ──▶ Provisioning ──── pod /v1/models 200 ──▶ Ready
Ready ──── !peakWindow.contains(now) ──▶ Draining ──── inflight==0 OR grace_s elapsed ──▶ Destroying
Destroying ──── Vast destroy ack ──▶ Asleep
Provisioning ──── cold-start budget timeout ──▶ Asleep + Sentry alert
Provisioning ──── schedule out of window during cold-start ──▶ Destroying (cancel-in-flight)
```

**What:** Current `upstreams.Loader.tier0Override` map só tem chave `"llm"` (emergency pod = LLM-only per Phase 6.5 D-C2). Primary pod cobre **3 roles**: llm + stt + embed. Need to extend the map at construction time.

**When to use:** Primary pod healthy transition.

**Example:**

### Anti-Patterns to Avoid

- **Reusing health-bridge:** D-05 explicitly DROPS it. Gateway probe loop (Phase 3) does per-upstream `/health` HTTP probes every 10s, with breaker FSM tracking failures. Adding back a health-bridge sidecar duplicates this signal.
- **`fmt.Sprintf` inside the onstart Go const:** Phase 6 RESEARCH.md Pitfall 9 enforced — shell quoting is treacherous. Pass dynamic values via `Env` map; onstart reads via `$VAR` (bash expansion).
- **Reading schedule from DB at runtime:** D-08.1 + user explicit "via env vars". Schedule mudança = restart gateway. Don't add a `primary_schedules` Postgres table.
- **Letting cold-start budget block schedule-down:** If schedule down fires while still Provisioning, must cancel-in-flight (analog of emerg `cancelActiveLifecycle` from `lifecycle.go:1001`). Don't ride out 20min cold-start just to immediately destroy.
- **Putting llama-server in background + supervisord controlling 4 procs:** PID 1 should still be llama-server (crash detection — Phase 6 Pitfall 3). If supervisord = PID 1, llama-server crash doesn't kill the container; reconciler thinks pod is healthy. **`exec` llama-server at end of onstart, like emerg does.**

## Common Pitfalls

### Pitfall #1: Onstart script size approaching 16KB Vast.ai limit (CARRIED + AMPLIFIED from Phase 6 Pitfall 4)

**What goes wrong:** Vast.ai onstart_cmd hard limit is **16KB** `[VERIFIED: Vast.ai docs faq.cdn.vast.ai]`. Phase 6 emergencyOnstart is ~1970 chars (1.9KB). Phase 6.6 primary onstart with 4 services + weights fetch + supervisor logic could realistically be 3.5-5KB. Still comfortable.

**Why it happens:** Adding service-start lines, multiple weight downloads, error handling.

**How to avoid:**
- Build the onstart const in `primary/onstart.go` as a single raw-string with carefully-curated content.
- Run `len(buildPrimaryOnstart(...))` as a Go test assertion: `require.Less(t, l, 14000, "leave 2KB margin under Vast 16KB hard limit")`.
- If approaching limit, push complexity into a fetched-from-MinIO script (analog to Phase 6 D-04 original — abandoned because emerg fit inline). Trade-off: extra fetch latency + sha256 verify for the script itself.

**Warning signs:** PR review comment "this onstart is getting really long". Test `TestPrimaryOnstartLengthBelowLimit` fails.

### Pitfall #2: PID 1 wrong (multi-service in container)

**What goes wrong:** If 4 services started + supervisord (or `wait`-loop) becomes PID 1, then llama-server crash doesn't terminate the container. Vast `actual_status` stays `running`. Reconciler thinks pod is healthy. Requests fail with 502 because llama-server is dead but pod URL is still in tier-0 override.

**Why it happens:** Whoever owns "the final exec" gets PID 1. In Phase 6 emergency pod (1 service), `exec /app/llama-server` is the last line — llama-server IS PID 1. Crash → container dies → Vast flips `actual_status=exited` → reconciler triggers (Phase 6 Pitfall 3).

In Phase 6.6 with 4 services, if we do `supervisord -n` as the final line, supervisord owns PID 1.

**How to avoid:** **Keep `exec /app/llama-server` as the final line of the onstart bash, AFTER the other 3 services are spawned in background.** Speaches, Infinity, dcgm-exporter are children of bash subshell; llama-server replaces bash as PID 1 via `exec`. If llama-server crashes → container terminates → Phase 6.6 reconciler's Phase 3 probe detects breaker OPEN → schedule reconciler issues destroy on next tick.

**Trade-off:** Speaches/Infinity/DCGM crash does NOT crash the pod. Their `/health` endpoints will fail; Phase 3 breaker for those upstreams will open; gateway falls back to OpenAI Whisper / OpenAI embed. ACCEPTABLE per Phase 3 design.

**Warning signs:** Manual SSH inspection of pod shows `ps -ef` with llama-server NOT as PID 1.

### Pitfall #3: Cold-start delays (carried from Phase 6 SC-2 gap)

**What goes wrong:** Phase 6 lifecycle 39 measured 20m 58s cold-start vs 6min target. Root cause: 16.74 GB Qwen GGUF download Hetzner DE → Spain ES @ 14 MB/s. Phase 6.6 has THREE downloads:
- Qwen3.6-27B Q4_K_M (17.1 GB)
- Whisper large-v3 (~3 GB tarball, then untar)
- BGE-M3 (~1.5 GB tarball, then untar)
**Total: ~21.6 GB download → ~28-35 min cold-start over WAN.**

**Why it happens:** No persistent volume + no host pinning + cross-continent WAN.

**How to avoid:**
- **Parallel mc cp**: download 3 files concurrently via `mc cp ... & mc cp ... & wait`. Aggregate throughput limited by Vast inet_down (filter `≥500 Mbps` per `vast.DefaultSearchFilter` from `vast/types.go:188`). At 500 Mbps = 62.5 MB/s aggregate; if sustained, 21.6GB in ~6min. Reality is closer to 14 MB/s observed — so 25-30 min realistic.
- **Schedule pre-warm**: D-09.1 already addresses this — `PRIMARY_POD_SCHEDULE_UP_HOUR=7` if peak starts at 8h means pod is ready by 7:25-7:30. Operator orchestrates.
- **Accept the gap**: same SC-2 gap as Phase 6; documented in 06.6-VERIFICATION as "perf optimization tracked for future phase".

**Warning signs:** Operator complaints that pod isn't ready when peak hour starts.

### Pitfall #4: Schedule rule timezone mistakes (DST / boundary)

**What goes wrong:** Schedule `PRIMARY_POD_SCHEDULE_TIMEZONE=America/Sao_Paulo` + UP_HOUR=8 + DOWN_HOUR=22 means "08:00 BRT → 22:00 BRT". Brazil DST was abolished in 2019 — no longer relevant. BUT if operator deploys with `UTC` timezone by mistake, the pod is up 11h late.

**Why it happens:** Default timezone in container is UTC; `time.LoadLocation` panics if tz database missing (some scratch/distroless images).

**How to avoid:**
- Default `PRIMARY_POD_SCHEDULE_TIMEZONE=America/Sao_Paulo` in `config.go`.
- At Reconciler startup, `time.LoadLocation(cfg.PrimaryPodScheduleTimezone)` — if returns error, **fail-fast at gateway boot** (vs warn-log + use UTC). Operator must fix env var.
- Log the resolved next up-transition and next down-transition timestamps at INFO so operator can sanity-check.
- Unit tests must cover: UTC vs BRT, Saturday/Sunday excluded if `SCHEDULE_DAYS=mon,tue,wed,thu,fri`, midnight rollover (if DOWN_HOUR=22 + UP_HOUR=8, the "off" window is 22:00-08:00 of next day).

**Warning signs:** `gatewayctl primary schedule show` displays nonsensical next-transition.

### Pitfall #5: Qwen3.6 Jinja template incompatibility (NEW)

**What goes wrong:** Qwen3.6 model card explicitly documents new chat template features: `enable_thinking`, `preserve_thinking`, `developer` role support, `qwen3_coder` tool-call parser. The existing `pod/templates/qwen3.5-27b-tool-calling.jinja` (sha256 `1067302cc6d927210a84775b9a060f724da15debc168c79710cbf763512e9f67`) is Qwen 3.5-era and is the community-patched version. Running Qwen3.6 with the Qwen3.5 Jinja **likely** causes:
- Tool-calls returning malformed JSON arguments (issue #20198 in ggml-org/llama.cpp `[CITED: github.com/ggml-org/llama.cpp/issues/20198]`)
- `<think>...</think>` tags appearing as plain text in completion (template doesn't extract them into `reasoning_content`)
- developer-role messages mishandled

**Why it happens:** Template patches were Qwen3.5-specific. Qwen3.6 deserves its own.

**How to avoid:** 3 options ranked by recommendation:
1. **RECOMMENDED: Fetch Qwen3.6 native chat template from upstream** — the Qwen3.6 GGUF likely embeds chat_template via metadata (llama.cpp `--jinja` reads it automatically). Run a spike: provision pod with `--jinja` ONLY (no `--chat-template-file`), test tool-calling. If GGUF-embedded template works → omit `--chat-template-file` entirely, simpler config. Plan 06.6-01 task: spike.
2. **Fallback: Upload Qwen3.6-specific Jinja to MinIO** at `qwen-templates/qwen3.6-27b-tool-calling-<sha>.jinja`, pattern from Phase 6 D-04-B B2. Operator extracts from `Qwen/Qwen3.6-27B` HF repo (`chat_template.jinja` file). Onstart fetches with sha256 verify.
3. **Worst case: Keep Qwen3.5 Jinja** and accept reduced tool-calling/thinking compat. Validate which features still work. Document degraded mode in RUNBOOK.

**Warning signs:** `/v1/chat/completions` with `tools=[...]` returns malformed arguments or `<think>` leaks into content.

### Pitfall #6: HF model auto-download in Speaches/Infinity (cold-start surprise)

**What goes wrong:** Speaches and Infinity both default to **downloading models from HuggingFace at first start** (via `HF_HOME` cache dir). If MinIO weight pre-fetch is for Whisper/BGE-M3 BUT Speaches/Infinity ignore `/weights/*` and try to download from HF instead, we get:
- Extra ~3GB Whisper download from HF
- Extra ~1.5GB BGE-M3 download from HF
- Cold-start adds 10-20 min on top of MinIO fetch (if HF rate-limited or slow)
- Network egress to HuggingFace may be 429-blocked on Vast spot hosts (rare but observed)

**Why it happens:** Mount semantics. Phase 1 docker-compose.yml does `${WEIGHTS_DIR:-/weights}/whisper:/weights/whisper:rw` for speaches with `WHISPER_MODEL_PATH=/weights/whisper`. For Infinity does `${WEIGHTS_DIR:-/weights}/bge-m3:/app/.cache/huggingface:rw`. Direct dockerd doesn't exist in args runtype.

**How to avoid:**
- **Option A (DinD path):** keep docker-compose.yml semantics (volume mounts). Bind `/weights/whisper` to `/weights/whisper` inside speaches container; bind `/weights/bge-m3` to `/app/.cache/huggingface` inside infinity container. Set `WHISPER_MODEL_PATH=/weights/whisper` and `HF_HOME=/app/.cache/huggingface`. Requires privileged Vast container — uncertain (see §Decisions Resolved item 1 + Pitfall #7).
- **Option B (native binary path):** speaches and infinity don't ship standalone binaries — they are Python apps in Docker. Native binary path is NOT viable for these two services without significant work.
- **Option C (DinD-lite via `nerdctl` or `podman`):** rootless containers inside the outer Vast container. UNTESTED — risky.
- **PRACTICAL Option D (RECOMMENDED for primary pod):** **DinD** is the only viable orchestration if STT/embed must work. Run `dockerd` in background inside the Vast outer container, then `docker compose up` the 4 services. The Phase 1 `pod/docker-compose.yml` is REUSABLE this way (only delete health-bridge service definition). Requires Vast offer with privileged container support OR `--privileged` permission. **THIS IS A MAJOR FINDING — see §Decisions Resolved item 1 below.**

**Warning signs:** Speaches log "Downloading model.bin... 1/1500MB". Infinity log "Loading BAAI/bge-m3... 1/500MB".

### Pitfall #7: Vast.ai `--privileged` container support unverified

**What goes wrong:** DinD needs `--privileged`. Vast.ai create_instance API doesn't document a `privileged` JSON field. Vast offers vary by host configuration; some hosts may not permit privileged containers (security policy).

**Why it happens:** Vast.ai security model — hosts opt-in to capabilities.

**How to avoid:** Plan 06.6-01 Wave 0 task: **SPIKE** — provision 1 Vast pod (4090, ~5min @ $0.30, ~$0.03 cost) with `Env` containing `"--privileged": "1"` (analog to existing `"-p PORT:PORT": "1"` Docker forwarding convention from `lifecycle.go:780`) AND run `docker info | head -3` in onstart to verify DinD works. If privileged not supported, fallback strategies are:
  - (a) Use Vast offers with `verified=true` filter (already in `DefaultSearchFilter`) + check `host_capabilities` field if exists
  - (b) Static-binary path for Whisper (use whisper.cpp standalone) + Infinity (use infinity-py + `--engine ctranslate2`)
  - (c) **Reduce primary pod to LLM-only** (same as emergency) and accept STT/embed always run on tier-1 fallback (OpenAI). This drops user value but is the safest implementation.

**Warning signs:** Spike pod with `--privileged` env returns "permission denied" when DinD starts dockerd.

### Pitfall #8: `--ports` field omission breaks tier-0 override

**What goes wrong:** Strategy B uses `Env` map to forward Docker `-p PORT:PORT` flags (e.g., `"Env": {"-p 8000:8000": "1"}` per `lifecycle.go:779`). Primary pod needs 4 ports: 8000 (LLM), 8001 (STT), 8002 (embed), 9400 (DCGM). Forgetting one means gateway probe loop can't reach that service → breaker stays OPEN → tier-0 override fails for that role.

**Why it happens:** Manual env construction, easy to miss a port.

**How to avoid:** Build `Env` map via Go helper that returns a fixed structure:

```go
// File: gateway/internal/primary/lifecycle.go
func buildPrimaryEnv(cfg config.Config) map[string]string {

        "-p 9400:9400": "1",  // DCGM
        // MinIO + weights (same as emerg pattern)
        "MINIO_ENDPOINT": cfg.MinioEndpoint,
        "MINIO_BUCKET": cfg.MinioBucket,
        ...

        env["-p 22:22"] = "1"  // SSH debug (optional)

    }
    return env
}
```

Unit test asserts the 4 port entries are present.

**Warning signs:** Gateway log "STT upstream breaker OPEN" after primary marked Ready.

### Pitfall #10 — DRAIN STATE state-machine error (NEW)

**What goes wrong:** Schedule-down fires while pod has 50 inflight LLM requests. If reconciler immediately DestroyInstance, those 50 requests get 502. If reconciler waits forever for inflight=0, never destroys (peak load = persistent low-level inflight).

**How to avoid:** `PRIMARY_POD_SCHEDULE_GRACE_RAMP_DOWN_SECONDS` (default 300s = 5min). On schedule-down:
1. `RestoreTier0("llm"|"stt"|"embed")` — new requests immediately route to fallback chain
2. Start drain timer
3. Every 10s, check inflight count via dispatcher (Phase 5 inflight registry; `gateway/internal/shed/...`). If inflight == 0, transition immediately to Destroying. Else wait until timer elapses.
4. After grace_s, DestroyInstance regardless of inflight (accept that some long-running requests get truncated).

**Sentry breadcrumb:** emit `primary_drain_started` with inflight_count at start + `primary_drain_completed` with inflight_count at destroy time.

**Warning signs:** Primary pod destroyed mid-request → user-facing 502 errors. Or pod never destroyed → cost overrun.

### Pitfall #11 — Reconciler race: simultaneous emerg + primary provision (NEW)

**What goes wrong:** Primary pod scheduled UP at 08:00. Emergency reconciler already provisioned emergency pod at 07:55 because primary was Asleep (and local-llm breaker therefore OPEN since 06:00). At 08:00, primary reconciler provisions a NEW primary pod. Now we have 2 Vast pods running for 20min (cold-start) until primary marks Ready — **double cost ~$0.86/h × 0.33h = $0.29 waste per double-provision day**.

**How to avoid:** Primary reconciler's evaluateTick `Asleep → Provisioning` transition should INSPECT the emerg FSM state. If `emerg.FSM.State() == EmergencyActive`, the primary reconciler should:
- (Option A) Skip provisioning this peak window. Operator alerted via Sentry "primary skipped due to emerg active".
- (Option B) Provision primary anyway, and on primary Ready, fire a Pub/Sub event that triggers emerg `cancelActiveLifecycle` / destroy.
- (Option C) Coordinate via a Redis flag `gw:provisioning:lock` shared between emerg + primary reconcilers — only one may transition to *Provisioning at a time.

**RECOMMENDATION: Option B** — primary takes over (it's the intended source), emerg gracefully shuts down once primary serves. Maintains "failover invisible" core value. Implementation: extend `gateway/internal/emerg/reconciler.go` to subscribe to a new `gw:primary:events` channel, and on `type=primary_ready` event, force-destroy if EmergencyActive.

**Warning signs:** Cost dashboard shows 2 Vast pods for 20-30 minutes daily at 08:00.

### Pitfall #12 — Long-running pod cost monitoring drift (NEW)

**What goes wrong:** Schedule keeps pod up 14h/day × 22 weekdays × $0.40/h = **$123.20/month**. The existing `MonthlyEmergencyBudgetBRL` (default 200 BRL = ~$40 USD) is dimensioned for emergency pods (rare events). Primary pod regular use **will exceed this monthly easily** every single month. Sentry alerts fire daily.

**How to avoid:** Two separate budgets:
- `MonthlyEmergencyBudgetBRL` — unchanged (emergency events)
- NEW `MonthlyPrimaryBudgetBRL` — default 800 BRL (~$160 USD; gives 30% headroom over $123 nominal)

Audit cost separately: `audit_log` `event_kind` extended with `primary_lifecycle_close` (analog to `emergency_lifecycle_close`). Calculate primary cost as `dph × hours_active × USDToBRLRate` (same formula as emerg `calculateCostBRL` at `lifecycle.go:532`).

**Warning signs:** Same Sentry budget alert fires 30× per month.

### Pitfall #13 — POD-04 VRAM ≤21GB ceiling reduced by 4-service load

### Item 10 — Drain ramp-down implementation

**Evidence:** Phase 3 breaker FSM has 3 states: closed / half-open / open. There is NO `draining` state in breaker — that's a Phase 6.6 concern at the orchestration layer, not breaker.

**Recommendation:** Drain is implemented in `primary.Reconciler.evaluateDraining`:

```go
// pseudocode
func (r *Reconciler) evaluateDraining(ctx context.Context, now time.Time) {
    // Already called RestoreTier0 on transition to Draining.
    // New requests now route to tier-1 (OpenRouter/OpenAI) per fallback chain.
    
    inflight := r.dispatcher.InflightForRole("local-llm") +
                r.dispatcher.InflightForRole("local-stt") +
                r.dispatcher.InflightForRole("local-embed")
    elapsed := now.Sub(r.drainStartedAt)
    grace := time.Duration(r.cfg.PrimaryPodScheduleGraceRampDownSeconds) * time.Second
    
    if inflight == 0 || elapsed >= grace {
        r.fsm.Transition(StateDraining, StateDestroying, now, "drain_complete")
    }
    // else: continue waiting; will re-check next tick (1Hz)
}
```

`r.dispatcher.InflightForRole(role)` is a new method to expose Phase 5 inflight counters per-role. The internal `shed.InflightRegistry` already tracks this — only need to add a public read method. Plan 06.6-04 (Wave 2) task: add `(*shed.InflightRegistry).Count(role string) int64`.

**No new breaker state needed.** Drain is a primary reconciler concern; breaker continues to do its job (health probe shows pod healthy → CLOSED → if we don't redirect via OverrideTier0/RestoreTier0, requests would still hit pod). The drain mechanism is the RestoreTier0 (cuts off new requests) + reading inflight to know when destroy is safe.

## Schedule Reconciler Design

### Go module layout

```
gateway/internal/primary/
├── fsm.go           # 5-state atomic FSM
├── schedule.go      # ScheduleRule struct + IsInPeak(now)
├── reconciler.go    # evaluateTick + state branches
├── lifecycle.go     # buildCreateRequest + provisionLifecycle + cancelActiveLifecycle
├── onstart.go       # raw-string Go const + buildPrimaryOnstart
├── tracker.go       # (optional) per-replica primary state mirror via Pub/Sub
└── *_test.go
```

### 5-state FSM

```
   ┌──────────┐                                              ┌────────────┐
   │  Asleep  │ ─── schedule.IsInPeak(now) ──────────────▶  │Provisioning│
   └──────────┘                                              └────────────┘
        ▲                                                          │
        │                                                          │ pod /v1/models 200
        │                                                          │ AND /health 200 (stt+embed)
        │                              ┌────────────┐              │ AND /metrics 200 (dcgm)
        │   DestroyInstance ack ────── │ Destroying │ ◀──────┐     ▼
        │                              └────────────┘        │  ┌───────┐
        │                                                    │  │ Ready │
        │   drain_complete ──┐         ┌────────────┐        │  └───────┘
        │                    └──────▶  │  Draining  │ ◀──────┘     │
        │                              └────────────┘              │ !schedule.IsInPeak(now)
        │                                                          │
        │   cold-start budget timeout ──── (Sentry alert) ◀────────┘
        │                                                          
        │   cancel-in-flight (schedule down during provisioning) ──┐
        └──────────────────────────────────────────────────────────┘
```

| State | Entry trigger | Exit trigger(s) | Side effects on entry |
|-------|---------------|------------------|------------------------|
| Asleep | Boot OR Destroying complete | `schedule.IsInPeak(now)` AND `!cfg.PrimaryPodScheduleDisabled` | None |
| Provisioning | Asleep transition | All 3 health probes pass → Ready; OR cold-start budget timeout (40min) → Asleep + Sentry; OR schedule out of window → Draining (cancel-in-flight) | `vast.CreateInstance(payload)`, store `activeLifecycle` |
| Ready | Provisioning success | `!schedule.IsInPeak(now)` → Draining; OR pod becomes unhealthy (all 3 breakers OPEN > 60s) → Draining | `loader.OverrideTier0("llm"/"stt"/"embed", podURLs)`, `dcgm.Scraper.SetURL(pod_9400)`, NOTIFY upstream change |
| Draining | Ready transition | `inflight == 0` OR `grace_s` elapsed → Destroying | `loader.RestoreTier0("llm"/"stt"/"embed")`, `dcgm.Scraper.SetURL("")` (or default), start grace timer |
| Destroying | Draining complete | `vast.DestroyInstance(id)` ack → Asleep | `bestEffortDestroy`, close lifecycle row, clear `activeLifecycle` |

### Env var contract

See §Decisions Resolved item 5 for the 22 env vars.

### Integration with breaker FSM

**No new breaker states.** Primary reconciler reads breaker state via existing `breaker.Set.State(name)` API. The pod's 3 service breakers (`local-llm`, `local-stt`, `local-embed`) are populated by gateway's Phase 3 probe loop as soon as the upstream URLs are added via `OverrideTier0`.

**Sequence on Ready transition:**
1. Pod `/v1/models` returns 200 → primary reconciler considers pod alive
2. `OverrideTier0("llm", podURL)` → Phase 3 probe starts probing tier-0 LLM
3. Within 10s (next probe tick) breaker `local-llm` transitions to CLOSED
4. Similar for STT + embed
5. Primary FSM transitions Provisioning → Ready

**Sequence on Draining:**
1. Primary FSM detects schedule out of window
2. Primary FSM `RestoreTier0("llm"/"stt"/"embed")` IMMEDIATELY → new requests route to tier-1 (OpenRouter/OpenAI)
3. Primary FSM enters Draining state
4. Existing in-flight requests continue to use original ReverseProxy directors (they hold a reference to the URL at dispatch time)
5. Reconciler polls `dispatcher.InflightForRole(role)` every tick
6. When inflight == 0 OR grace_s elapsed → Destroying

### Leader election

**Recommendation: USE redsync leader election** (parity with emerg). Two gateway replicas running same schedule would both try to provision a pod — Vast.ai accepts both, doubles cost. Leader-elected ensures only one tries.

Reuse `redisx.NewEmergRedsync` pattern; new key `gw:primary:lock` (separate namespace from emerg).

**Alternative:** every replica evaluates schedule + checks `gw:primary:lifecycle:active` flag before CreateInstance. Simpler but race-prone (TOCTOU). Reject.
