<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Next.js server | Client components run in the operator's browser; must not carry secrets |
| Next.js client → /api/gateway/* | All gateway data flows through the server-side proxy from 07-07 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-29 | Information Disclosure | client components fetching the gateway | mitigate | Every component fetches via the `dashboard/src/lib/gateway.ts` wrappers, which call the `/api/gateway/*` server proxy (07-07) — never the gateway directly, never the admin key. The Task 4 checkpoint explicitly verifies in devtools that no browser request carries `X-Admin-Key`. |
| T-07-30 | Tampering | "Reconhecer incidente" action | accept | UI-SPEC locks the dashboard as read-only — "Reconhecer incidente" mutates only local component state (silences the banner for 5 min, mirroring the alerting dedup TTL). It issues no gateway write, so there is no tamper surface; an acceptance criterion greps the banner for any fetch/mutation call. |
| T-07-31 | Information Disclosure | audit-log content in the UI | mitigate | The audit table renders only the metadata columns the 07-03 `/admin/audit` handler returns (ts, route, status, latency, event_kind) — `audit_log_content` (prompts/responses) is never in the response, so sensitive tenant content cannot reach the browser. |
| T-07-32 | Spoofing | direct navigation to a dashboard route | mitigate | The `(dashboard)` route group sits behind the 07-07 `middleware.ts` session gate; an unauthenticated direct navigation is redirected to `/login` before any component renders. |
</threat_model>
