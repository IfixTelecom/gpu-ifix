<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Redis Pub/Sub → alerter | Event payloads (possibly malformed) cross into the alerting path |
| alerter → external channels | Severity classification decides which untrusted-by-default operators get paged |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-15 | Denial of Service | alerter consume loop | mitigate | Pitfall 5 — the consume loop classifies + dedups + ENQUEUES to bounded per-channel workers; it never calls `channel.Send` synchronously. A dead Chatwoot cannot stall event consumption. A full worker queue increments `gateway_alert_dropped_total` and the loop continues. A test asserts a blocking `Send` does not stall classification. |
| T-07-16 | Denial of Service | alert fatigue / flooding | mitigate | OBS-06 — `dedupShouldSend` does `SET NX EX 300`; a repeated event within 5 minutes skips all external channels (still logged). One incident = one page, not ten. |
| T-07-17 | Tampering | malformed Pub/Sub payload | mitigate | `severityFor` returns an error for unparseable JSON; the alerter logs a WARN and `continue`s — a malformed (or hostile) payload cannot crash the goroutine or be reflected into a `Send`. Canonical `breaker/subscribe.go` pattern. |
| T-07-18 | Repudiation | missed critical during boot/restart | mitigate | Pitfall 4 — `ReconcileBoot` reads the current FSM/breaker state on startup so a gateway restart during an active incident still raises the alert; the alerter goroutine is spawned EARLY (wiring enforced in plan 07-06). |
| T-07-19 | Information Disclosure | dedup fail-mode | accept | On a Redis error, `dedupShouldSend` fails OPEN for `critical` (a duplicate page is acceptable; a missed page is not) and CLOSED for `warning`/`info`. Documented tradeoff from RESEARCH.md Pattern 2. |
</threat_model>
