The three outbound alert clients + the Redis alert-helper module. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| gateway process → Chatwoot / ClickUp / Brevo (external) | Outbound HTTP/SMTP carrying API tokens crosses the process boundary |
| external API → gateway | Untrusted response bodies + headers are parsed back in |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-11 | Information Disclosure | alert client errors / logs | mitigate | Each client's API token touches an `*http.Request` in exactly ONE greppable `setAuthHeader`-style method (verified: `grep -c` returns 1). Errors are package sentinels mapped from status code — they never embed the URL, token, or any header. Bounded `io.LimitReader(resp.Body, 16*1024)` prevents a hostile response from ballooning an error string. |
| T-07-12 | Denial of Service | slow/dead external API | mitigate | Each client has a fixed package-level `httpTimeout` const (no unbounded wait) AND its own `gobreaker.CircuitBreaker` — a dead Chatwoot trips its breaker and fails fast instead of stalling. (The alerter's bounded-worker isolation is plan 07-05; this plan ensures the per-client fast-fail.) |
| T-07-13 | Spoofing | ClickUp 401 retry storm | mitigate | Pitfall 6 — a 4xx-except-429 (incl. a 401 from a bad/rotated token) returns `backoff.Permanent`, stopping the retry immediately. A test asserts exactly one server hit for a 401. No credential-guessing amplification. |
| T-07-14 | Tampering | untrusted external response body | mitigate | Response bodies are read through `io.LimitReader` and only the status code + a bounded error envelope are interpreted; no response field is reflected into a privileged operation. |
</threat_model>
